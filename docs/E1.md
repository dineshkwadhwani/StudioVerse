# E1 — App Skeleton + Mobile Wrapper

Status: Working markdown version for Copilot context.  
Canonical source: Formal E1 specification document.

## Implementation status snapshot — April 12, 2026

- The authenticated Coaching Studio shell is implemented with role-aware dashboard entry, logged-in dropdown navigation, and responsive page headers.
- Additional authenticated routes now exist beyond placeholders, including My activities and Manage Wallet.
- Mobile-wrapper readiness remains preparatory; the current implementation focus is the web app shell and route/menu foundations.

## Objective

Build the shared application skeleton for StudioVerse and prepare the platform for mobile-wrapper usage.

E1 establishes the base app shell that authenticated users will eventually work inside. It provides the common layout, placeholder screens, route structure, role-aware menu foundations, dashboard entry points, mobile navigation behavior, and Capacitor readiness.

## Scope

Epic E1 covers:
- shared app layout
- placeholder pages
- app routing structure
- role-aware menu foundations
- dashboard placeholders
- mobile UI shell behavior
- Capacitor integration baseline
- mobile navigation
- build readiness
- environment configuration for app-shell usage

This epic is about the app framework, not full business feature delivery.

## Prerequisites

Assume T0, T1, T2, T3, and T4 are complete enough to support:
- repo and environment setup
- frontend architecture
- Firebase baseline
- deployment conventions
- security and access-control baseline

E1 should build on the existing App Router structure and the shared StudioVerse architecture.

## Purpose

E1 creates the shared authenticated shell for the product.

It should ensure that:
- the application has a stable layout foundation
- major routes exist
- placeholder screens are available
- dashboard paths are ready
- mobile navigation patterns are in place
- the project is ready for Capacitor wrapping later
- environment config supports the app shell correctly

## In scope

E1 includes:

## Out of scope

Do not implement the following in E1:
- final authentication flows
- production user registration/login behavior
- real dashboard widgets
- business data integrations
- program execution
- tool execution
- reports generation
- admin workflows
- full role enforcement logic beyond skeleton readiness
- full mobile native feature implementation

E1 should prepare the structure for these features, not complete them.

## App shell expectations

The app shell should:
- use shared layouts
- support desktop and mobile behavior
- provide a stable header/sidebar/navigation pattern
- provide placeholder content regions
- support future auth guards
- support future role-aware navigation
- be extendable without reworking the route foundation

## Layout requirements

The app should include:
- root layout
- authenticated app layout
- stable header area
- sidebar or equivalent desktop navigation
- main content area
- responsive mobile navigation behavior

General layout rules:
- keep layout modular
- separate shell components from business modules
- preserve future role-driven menu behavior
- avoid hardcoding business feature logic into the shell

## Placeholder screens

Create placeholder pages for key routes so the structure exists before feature implementation.

These placeholders should:
- render cleanly
- identify the screen purpose
- avoid broken navigation
- be minimal but production-safe
- make later feature replacement easy

Placeholder pages should exist for the main app areas that later epics depend on.

## Routing foundation

E1 should establish the authenticated route skeleton for areas such as:
- dashboard
- programs
- tools
- events
- cohorts
- assignments
- reports
- notifications
- settings
- admin

Rules:
- route naming should align with the documented architecture
- placeholder pages should compile cleanly
- future guarded routes should fit naturally into the structure
- avoid route patterns that will require later rework

## Menu foundations

Prepare the menu structure for role-aware rendering later.

This means:
- define navigation groups
- define menu labels
- define route targets
- prepare a clean structure for role filtering later
- do not hardcode unstable ad hoc menu logic across multiple components

Roles and visibility logic do not need to be fully implemented yet, but the menu architecture should anticipate:
- super admin
- company
- professional / coach
- individual

## Dashboard placeholders

Create dashboard entry routes and placeholder screens for the major user types.

At minimum, prepare for:
- admin dashboard
- company dashboard
- professional / coach dashboard
- individual dashboard

These can be placeholder shells for now, but the route structure should be ready for later dashboard implementation.

## Mobile UI expectations

E1 should support responsive app-shell behavior beyond the public landing page.

Mobile expectations:
- shell adapts to smaller screens
- mobile nav is usable
- touch targets are appropriate
- sidebar behavior collapses or transforms cleanly
- layout remains stable across common device sizes

This is not full native behavior yet, but it should be mobile-conscious and wrapper-friendly.

## Mobile navigation

Prepare mobile navigation patterns such as:
- menu trigger
- drawer, sheet, or bottom navigation depending on current shell design
- clean open/close behavior
- accessible focus handling
- predictable route access on mobile

Keep the mobile navigation implementation lightweight and extendable.

## Capacitor baseline

E1 should prepare the project for mobile wrapping with Capacitor.

This includes:
- ensuring app-shell routing works cleanly in a wrapper-compatible way
- confirming build output is compatible with the current approach
- preparing the codebase for Capacitor config and later platform sync
- avoiding web-only assumptions that would block mobile wrapping

This does not require full native feature integration in E1.

## Build requirements

E1 should build cleanly with the app skeleton in place.

Validate:
- no broken imports
- no dead placeholder navigation
- no route conflicts
- no layout compilation failures
- no environment config issues related to shell behavior

The app should remain deployable after adding the skeleton.

## Environment configuration

E1 should align the app shell with the environment model already established in the technical setup.

Environment behavior should:
- support dev / staging / prod conventions
- respect public vs server-side config boundaries
- avoid exposing secrets
- support the app-shell use case cleanly
- remain consistent with the Firebase and Vercel architecture

## Technical implementation rules

Use:
- Next.js App Router
- TypeScript
- existing folder structure
- shared module conventions
- architecture-aligned routing

Follow these rules:
- keep shell logic separate from business logic
- avoid direct backend logic inside placeholder UI
- do not add unstable shortcuts that break later epic implementation
- preserve the multi-tenant structure
- keep role-aware behavior extensible
- keep navigation config centralized where possible
