## 1. Purpose

This Epic defines the functional requirements for a reusable AI bot framework available across all StudioVerse tenants, with tenant-specific behavior, persona, and visibility controls.

The bot supports two modes:
- **Studio Bot**: answers tenant/platform questions from tenant documentation context.
- **Professional Bot**: answers generic domain/professional questions (for example coaching guidance in Coaching Studio).

The bot includes a guest onboarding flow, referral capture integration, configurable conversation limits for token/cost control, and tenant-level feature toggles managed from Tenant management.

## 2. Scope Summary

### In Scope
- Floating avatar launcher (bottom-right) for tenant pages.
- Tenant-configurable bot persona (name and avatar).
- Mode selector with two modes: Studio Bot and Professional Bot.
- Logged-in and guest-aware greeting and onboarding.
- Guest onboarding capture (name + phone, no OTP).
- Auto-referral creation for guest onboarding with platform Super Admin as referrer.
- Placeholder invite/notification behavior for referral flow.
- Conversation cap (default 5 user messages) for both modes.
- Post-cap follow-up behavior:
  - Logged-in users: no email ask; show "we will email you" message.
  - Guests: ask for email and update the same referral record.
- End conversation with thank-you after cap completion flow.
- Tenant-level configuration in existing `tenants` collection:
  - bot visibility
  - Studio Bot toggle
  - Professional Bot toggle
  - persona config
  - message cap
- Documentation retrieval corpus for Studio Bot from all `.md` files under `docs/`.

### Out of Scope
- PDF ingestion for bot knowledge (explicitly excluded for now).
- OTP verification in bot onboarding.
- Real email transport implementation (placeholder only in this phase).
- Voice bot / real-time audio avatar.
- Human handoff scheduler integration beyond stored follow-up capture.

## 3. Business Context

StudioVerse needs a reusable, tenant-configurable conversational assistant that can:
- reduce support burden for platform questions,
- offer domain guidance to prospective and current users,
- capture anonymous demand and convert guest users into leads,
- and control AI cost through strict message limits and context optimization.

## 4. Bot Modes

## 4.1 Studio Bot

**Definition:** Knowledge-grounded bot for tenant/platform questions.

**Behavior:**
- Uses tenant documentation context from `.md` files under `docs/`.
- Must answer from retrieved context only.
- If context is insufficient, bot should state that clearly and avoid hallucination.

**Primary Use Cases:**
- Feature explanations.
- Tenant process understanding.
- Navigation/help queries.
- Architecture/product behavior clarifications from docs.

## 4.2 Professional Bot

**Definition:** Generic expert professional assistant for the tenant domain.

**Behavior:**
- Uses LLM without tenant-doc retrieval dependency.
- Persona and tone remain tenant-configured.
- Responds as a domain expert (for Coaching Studio, coaching-oriented responses).

**Primary Use Cases:**
- Professional guidance.
- Conceptual Q&A.
- Lightweight advisory conversation.

## 6. User Journeys

## 6.1 Logged-in User Flow

1. User clicks avatar launcher.
2. Bot greets user by name.
3. User selects mode (Studio Bot or Professional Bot), subject to enabled toggles.
4. Conversation proceeds.
5. After message cap is reached:
   - Bot shows limitation message.
   - Bot states follow-up will be sent to registered email.
   - Conversation ends with thank-you.

## 6.2 Guest User Flow

1. User clicks avatar launcher.
2. Bot asks onboarding fields:
   - Name
   - Phone number
3. On capture completion:
   - Create referral record in `referrals`.
   - Referrer must be a platform Super Admin.
   - If multiple Super Admin users exist, earliest created Super Admin takes precedence.
   - Trigger placeholder invite/notification action.
4. User selects mode and starts conversation.
5. After message cap is reached:
   - Bot asks for email for follow-up.
   - Save email to the same referral record created at onboarding.
   - End with thank-you message.

---

## 7. Referral Integration Rules

## 7.1 Guest Referral Creation

When guest onboarding completes (name + phone), create a referral entry with:
- `tenantId`
- `referrerUserId` (oldest Super Admin)
- `referrerRole = superadmin`
- referred details from onboarding (`name`, `phone`)
- placeholder email if not yet available
- source marker indicating bot flow
- status initialized as referred

## 7.2 Email Update on Cap Completion (Guest)

When guest provides email after cap:
- update same referral record (`referredEmail`)
- do not create duplicate referral for the same bot session

## 7.3 Email Behavior (Logged-in)

For logged-in users post-cap:
- no email prompt
- show message: follow-up will be sent to registered email

## 8. Message Limit and Cost Control

Token/cost optimization is mandatory.

Baseline requirements:
- Enforce message cap for both Studio Bot and Professional Bot.
- Default cap is **5 user messages** per conversation.
- Cap value must be tenant-configurable (`messageCap`).
- Keep short recent history only (bounded turns).
- Use concise prompts and response length controls.

Studio Bot optimization requirements:
- Use retrieval with top relevant chunks only.
- Do not send full docs on each query.
- Context source: all `.md` files in `docs/`.
- Exclude PDFs in this phase.

Professional Bot optimization requirements:
- No heavy tenant context payload.
- Bounded history.
- Compact system prompt.

## 9. Tenant Configuration Requirements

Bot config must be stored in the existing tenant document (`tenants` collection), not a separate collection.

Suggested object:

```json
{
  "botConfig": {
    "visible": true,
    "studioBotEnabled": true,
    "professionalBotEnabled": true,
    "personaName": "Coach Dinesh",
    "personaAvatar": "/tenants/coaching-studio/bot.png",
    "messageCap": 5
  }
}
```

UI controls in Tenant Management:
- Toggle: Bot Visible
- Toggle: Studio Bot Enabled
- Toggle: Professional Bot Enabled
- Input: Persona Name
- Input: Message Cap
- Optional input: Persona Avatar path

## 12. Functional Acceptance Criteria

1. Bot visibility respects tenant toggle (`visible`).
2. If only one mode is enabled, bot directly starts that mode.
3. Logged-in users are greeted by name.
4. Guest users are required to enter name and phone before chat starts.
5. Guest onboarding creates exactly one referral record per session start.
6. Referrer assignment picks oldest Super Admin when multiple exist.
7. Studio Bot uses documentation retrieval from `.md` files only.
8. Professional Bot works without documentation retrieval.
9. Both modes enforce message cap.
10. Logged-in users at cap are not asked for email.
11. Guests at cap are asked for email and same referral is updated.
12. Conversation closes with thank-you after cap completion flow.
13. Placeholder invite/follow-up actions are triggered where specified.

---

## 13. Non-Functional Requirements

- Secure API key handling on server side only.
- Fast response target for standard queries.
- Graceful fallback message on AI provider failure.
- No leakage of hidden prompts or private credentials.
- Strict tenant scoping for retrieval context.

## 16. Risks and Mitigations

1. **Hallucinations in Studio Bot**  
Mitigation: strict retrieval grounding + explicit fallback when context not found.

2. **Token overrun/cost spikes**  
Mitigation: message cap, bounded history, top-k retrieval only.

3. **Lead quality issues from unverified phone**  
Mitigation: mark source as bot lead and allow downstream validation workflows.

4. **Duplicate guest records**  
Mitigation: session/browser correlation and same-record update for email capture.

## 18. Glossary

- **Studio Bot:** Documentation-grounded tenant support bot.
- **Professional Bot:** Generic domain expert bot.
- **Message Cap:** Maximum user messages allowed in a conversation session.
- **Guest Lead:** Unauthenticated user captured via bot onboarding.
- **Earliest Super Admin:** Super Admin user with earliest `createdAt`, used as default referrer for guest lead referrals.
