# TITLE StudioVerse - EPIC E13 - Buying Credits / Coin Packages

This Epic defines the functional requirements for selling credit packages inside StudioVerse. The capability includes a Super Admin package-management surface, a user-facing package purchase surface, order creation and tracking, placeholder payment simulation for success and failure outcomes, and wallet crediting upon successful payment.

Users in eligible roles can buy one package at a time through a simple transaction flow. Super Admin creates and maintains the packages, activates or deactivates them, and monitors all purchase orders through an Orders management page.

---

# TITLE StudioVerse - 1. Purpose

StudioVerse uses credits (coins) as a platform currency for usage flows such as assignments, programs, tools, events, and future monetized actions. The platform now needs a direct self-serve way for eligible users to buy additional credits.

The goal of this Epic is to enable:
- centrally managed credit packages,
- a clean user buying journey,
- order persistence and auditability,
- wallet top-up on successful payment,
- and a payment-ready architecture that can later plug into Razorpay without reworking the user flow.

# TITLE StudioVerse - 3. Business Context

The wallet already exists as a core construct in StudioVerse. Credits are used as a controlled internal economy for access, assignments, and monetized actions. Up to now, wallets have been funded mainly through onboarding grants and admin issuance.

This Epic introduces a direct monetization flow. It creates the first self-serve commerce capability in the platform while keeping complexity intentionally low. The design principle is to minimize friction: select one package, confirm summary, simulate payment outcome, then complete wallet top-up if successful.

This flow also creates the baseline architecture for future real-payment integration via Razorpay.

# TITLE StudioVerse - 6. Order Model

Each package purchase attempt must create or maintain an order record.

Suggested fields:

- `id`
- `userId`
- `walletId`
- `userRole`
- `studioType`
- `tenantId`
- `packageId`
- `packageNameSnapshot`
- `creditsSnapshot`
- `priceSnapshot`
- `currency`
- `status`
- `paymentProvider` default `razorpay_placeholder` in this phase
- `paymentReference` optional placeholder transaction id
- `paymentAttemptedAt`
- `paymentSucceededAt`
- `paymentFailedAt`
- `walletCredited` boolean
- `walletTransactionId` optional
- `createdAt`
- `updatedAt`
- `createdBy`

## 6.1 Order Statuses

Minimum statuses:
- `created`
- `pending`
- `paid`
- `failed`
- `cancelled` optional for future-proofing

Behavior rules:
- Order starts as `created` when the user confirms package selection.
- Enter payment flow; order becomes `pending` when user reaches placeholder payment page.
- On simulated success, order becomes `paid`.
- On simulated failure, order remains `pending` or moves to `failed` based on UX decision; for this Epic use `pending` on payment failure simulation only if the business wants retry potential, otherwise use `failed`.

### Final rule for this Epic

- Simulate Success -> set order status to `paid`.
- Simulate Failure -> set order status to `pending`.

This matches the requested requirement that failed payment simulation should leave the order visible as pending.

# TITLE StudioVerse - 12. User Flow - Placeholder Payment Screen

Real Razorpay integration is deferred in this phase. A payment placeholder screen must simulate the external payment experience.

## 12.1 Placeholder Screen Requirements

Screen shows:
- order summary,
- package name,
- credits,
- price,
- note that Razorpay integration is pending,
- two simulation buttons:
  - `Simulate Payment Success`
  - `Simulate Payment Failure`

## 12.2 Simulate Payment Success

On success:
- update order status to `paid`,
- generate placeholder payment reference,
- create wallet transaction credit entry,
- add coins to the user wallet,
- mark `walletCredited = true`,
- persist wallet transaction id into order,
- show success confirmation screen.

## 12.3 Simulate Payment Failure

On failure:
- keep order status as `pending`,
- do not credit wallet,
- do not create wallet credit transaction,
- show pending / payment not completed message,
- allow user to leave flow.

# TITLE StudioVerse - 13. Wallet Crediting Rules

Wallet changes must occur only after successful payment.

## 13.1 Success Rules

When payment is successful:
- wallet balance increases by package credits,
- wallet totalIssuedCoins increases if that is the chosen wallet accounting convention,
- wallet availableCoins increases by package credits,
- transaction ledger row is created,
- transaction reason should clearly identify package purchase.

Suggested transaction reason:
- `Credit package purchase - {packageName}`

Suggested transaction type:
- `credit`

## 13.2 Failure Rules

When payment fails or remains pending:
- wallet must not be changed,
- no credit transaction is written.

## 13.3 Idempotency Rule

Wallet crediting must be idempotent. Even if success endpoint or action is triggered twice accidentally, the wallet must be credited exactly once per paid order.

Implementation guidance:
- check `walletCredited` on order before wallet update,
- perform order+wallet+transaction update atomically where possible.

# TITLE StudioVerse - 21. Risks and Mitigations

1. Duplicate crediting on repeated success action  
Mitigation: idempotent order processing using `walletCredited` guard and atomic update.

2. Package edits corrupt historical order meaning  
Mitigation: snapshot package name, credits, and price on order creation.

3. Users confused by placeholder payment flow  
Mitigation: explicitly label simulation screen as temporary pre-integration step.

4. Admin package misconfiguration  
Mitigation: validation on credits, price, and active state; seed defaults for first launch.

5. Cross-role permission leakage  
Mitigation: enforce role checks on package admin routes, order admin routes, and purchase flow.
