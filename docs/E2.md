## 1. Purpose

This document defines the business and functional requirements for the Program Management and Tenant Catalog Publishing capability in StudioVerse.

The purpose of this Epic is to enable the SuperAdmin to create, manage, and publish Programs for a specific tenant so that those Programs become visible to all Companies, Professionals, and Individuals within that tenant. This Epic establishes the canonical Program entity for StudioVerse and creates the foundation for future Program-related capabilities such as assignment, consumption, progress tracking, catalog commerce, promoted content handling, and creator-led publishing workflows.

This PRD is intended to provide a shared understanding across business, product, engineering, QA, and integration stakeholders so that all teams align on scope, behavior, data design, lifecycle rules, and forward compatibility.

## 3. Executive Summary

Programs are a foundational content object in StudioVerse. They represent structured offerings that can later be assigned, consumed, tracked, and monetized. This Epic introduces the first governed Program management capability by enabling the SuperAdmin to create Programs directly into a tenant-wide catalog.

A Program created by the SuperAdmin is always created for one specific tenant. Once published and active, it is visible to all Companies, Professionals, and Individuals within that tenant. This creates a controlled, curated, and scalable catalog layer for each tenant.

This Epic also introduces schema elements required for future growth. In later phases, Companies and Professionals will be able to create their own Programs. However, those creator-owned Programs will initially remain scoped to the Company or Professional boundary and will not automatically become visible tenant-wide. If broader visibility is desired, the creator will submit the Program to the SuperAdmin for publication review through a future workflow.

In addition, this Epic introduces a new Program-level attribute called `promoted`. This attribute indicates that the Program has been marked for differentiated handling in future product behavior. The exact business behavior of promoted Programs is intentionally out of scope for this Epic and will be defined later together with the publish-request and broader catalog promotion workflow. In this Epic, the requirement is to ensure the data model, validation layer, and admin management flow can store and manage this attribute without requiring future schema redesign.

## 4. Business Context

StudioVerse is a multi-tenant platform serving different Studio contexts, such as Coaching Studio, Training Studio, and Recruitment Studio. Each tenant requires its own governed Program catalog so that learning and service-delivery assets can be curated and surfaced consistently across the tenant.

Programs are central to StudioVerse because they are expected to support:
- tenant-level catalog browsing,
- future assignment to Individuals and Cohorts,
- future progress tracking,
- future paid and free catalog offerings,
- future analytics,
- future promoted-program placement,
- and future creator-led publishing models.

Without a stable Program model and SuperAdmin-driven catalog creation capability, later product layers would be forced to build on fragmented assumptions or require costly redesign.

---

## 5. Problem Statement

StudioVerse currently needs a formal Program domain model and a controlled Program creation workflow.

The system must allow the SuperAdmin to create a Program for a selected tenant with the right metadata, lifecycle controls, availability windows, visibility behavior, and catalog-governance attributes. At the same time, the Program design must support future ownership models where Companies and Professionals can create Programs privately first and later request tenant-wide publication.

The model must also support a new `promoted` attribute so that future promotion-oriented behaviors can be built on top of the Program entity without schema changes.

Without this capability:
- there is no canonical Program structure,
- tenant-wide Program governance is missing,
- future catalog and assignment work lacks a stable dependency,
- future promotion workflows would require redesign,
- and future publication workflows would require schema migration or restructuring.

## 10. Domain Definitions

### 10.1 Tenant
A tenant is the top-level Studio context under which users, content, and catalog visibility are organized. Examples include Coaching Studio, Training Studio, and Recruitment Studio.

### 10.2 Program
A Program is a tenant-scoped content offering that may later be browsed, assigned, consumed, tracked, and monetized.

### 10.3 Ownership Scope
Ownership scope defines who created the Program and the default governance boundary of the Program.

Supported ownership scopes:
- `platform`
- `company`
- `professional`

This Epic implements `platform` creation only.

### 10.4 Catalog Visibility
Catalog visibility defines who can discover the Program.

Supported visibility scopes:
- `tenant_wide`
- `company_only`
- `professional_only`

This Epic implements tenant-wide visibility for SuperAdmin-created Programs.

### 10.5 Publication State
Publication state represents catalog-governance readiness and supports future approval workflows.

Supported values:
- `draft`
- `published`
- `pending_publication_review`
- `rejected_publication`

### 10.6 Status
Status represents the operational lifecycle of the Program.

Supported values:
- `draft`
- `published`
- `inactive`
- `archived`

### 10.7 Promoted
`promoted` is a Program-level attribute that indicates the Program has been marked for differentiated treatment in future catalog and publishing workflows.

In this Epic:
- `promoted` is stored as metadata only,
- it does not change visibility rules by itself,
- it does not automatically publish a Program,
- it does not automatically affect ranking or display,
- and it does not override tenant, publication, or lifecycle rules.

The functional meaning of promoted Programs will be defined in a future Epic together with the publish-request workflow.

## 12. Functional Requirements

### 12.1 Program Creation
The system shall allow the SuperAdmin to create a Program for a selected tenant.

The Program creation flow shall support capture of:
- Program name
- Program short description
- Program long description
- Program thumbnail
- Delivery type
- Duration
- Additional details
- Program video URL
- Credits required
- Program available date
- Program expiry date
- Status
- Facilitator name
- Promoted flag

### 12.2 Tenant Association
Every Program shall belong to exactly one tenant.

The tenant must be selected or resolved at the time of Program creation.

### 12.3 Facilitator Modeling
Facilitator shall be stored as plain text only.

No system-level user reference, relational link, or internal actor lookup is required for facilitator in this Epic.

### 12.4 Program Ownership
Programs created in this Epic shall be stored with:
- `ownershipScope = platform`
- `ownerEntityId = null`

The system shall also support future ownership scopes for:
- company-created Programs
- professional-created Programs

### 12.5 Catalog Visibility
Programs created by SuperAdmin shall default to tenant-wide catalog visibility when published.

Tenant-wide visibility means the Program is discoverable by:
- all Companies in the tenant,
- all Professionals in the tenant,
- all Individuals in the tenant.

### 12.6 Program Lifecycle
The system shall support the following status values:
- draft
- published
- inactive
- archived

The system shall also support the following publication states:
- draft
- published
- pending publication review
- rejected publication

### 12.7 Availability Windows
The Program may include:
- available from date
- expiry date

These dates shall govern end-user visibility in combination with status and publication state.

### 12.8 Program Editing
The system shall allow the SuperAdmin to edit existing Program metadata after creation.

This includes the ability to update the promoted flag.

### 12.9 Promoted Attribute
The system shall store a Program-level `promoted` attribute.

In this Epic:
- `promoted` shall be a boolean flag,
- it shall default to `false` unless explicitly set,
- it may be edited by SuperAdmin,
- and it shall be auditable like other important Program attributes.

### 12.10 Auditability
The system shall create audit records for key Program actions including:
- create
- update
- publish
- status change
- archive
- promoted flag change

## 14. Business Rules

- Only SuperAdmin can create Programs in this Epic.
- Every Program must belong to one and only one tenant.
- Program name is required.
- Program short description is required.
- Program long description is required.
- Delivery type is required.
- Duration value and duration unit are required.
- Credits required is required and must not be negative.
- Thumbnail is required for publish readiness.
- Video URL is optional.
- Facilitator name is optional and plain text only.
- Promoted is optional and defaults to `false`.
- Setting `promoted = true` does not by itself publish the Program.
- Setting `promoted = true` does not by itself make the Program visible.
- Setting `promoted = true` does not override tenant, publication, or date rules.
- Available date cannot be later than expiry date.
- Programs must not be hard deleted in MVP.
- Archived Programs remain in the system for governance and history.
- Published tenant-wide Programs are visible only within the tenant they belong to.

---

## 15. Visibility Rules

A Program shall be visible in the active tenant catalog only when all of the following are true:

- The Program belongs to the same tenant as the user.
- `status = published`
- `publicationState = published`
- Current date is on or after `availableFrom`, if specified.
- Current date is before `expiresAt`, if specified.

A Program shall not be visible in active tenant discovery if any of the following are true:

- `status = draft`
- `status = inactive`
- `status = archived`
- `publicationState != published`
- current date is before `availableFrom`
- current date is after `expiresAt`

The `promoted` flag shall not alter these base visibility rules in this Epic.

Administrative management views may display Programs regardless of end-user visibility state.

## 16. Future Compatibility Requirements

This Epic must support future extension to:

### 16.1 Company-Created Programs
Future Company-created Programs must be able to use the same Program schema while storing:
- `ownershipScope = company`
- `ownerEntityId = companyId`
- default `catalogVisibility = company_only`

### 16.2 Professional-Created Programs
Future Professional-created Programs must be able to use the same Program schema while storing:
- `ownershipScope = professional`
- `ownerEntityId = professionalId`
- default `catalogVisibility = professional_only`

### 16.3 Publication Request Workflow
A future Epic will introduce a publication request workflow whereby creator-owned Programs can be submitted for SuperAdmin review and approval before becoming tenant-wide.

The schema defined in this Epic must support that future workflow without redesign.

### 16.4 Promoted Program Workflow
A future Epic will define the functional meaning of `promoted = true`, including how promoted Programs are treated differently in catalog, publishing, ranking, placement, or discovery experiences.

The schema defined in this Epic must support that future workflow without redesign.

## 17. User Stories

### US-E2-01 — Define Program Domain Model
As a developer, I want a canonical tenant-aware Program model so that all Program features use one stable schema.

**Acceptance Criteria**
- Program schema supports tenant, ownership, visibility, publication, lifecycle, and promoted flag.
- Schema supports platform, company, and professional ownership.
- Facilitator is plain text only.
- Validation requirements are documented.
- Schema supports future publication review flows.
- Schema supports future promoted-program workflows.

### US-E2-02 — SuperAdmin Can Create a Draft Program
As a SuperAdmin, I want to create a Program in draft mode for a selected tenant so that I can prepare content before publishing it.

**Acceptance Criteria**
- Tenant is selected during create flow.
- Program draft can be saved successfully.
- Draft Program is stored as platform-owned.
- Draft Program is not visible in active tenant catalog.
- Promoted flag can be set or left unset during creation.

### US-E2-03 — SuperAdmin Can Add a Program Thumbnail
As a SuperAdmin, I want to attach a thumbnail to a Program so that it is visually represented in the catalog.

**Acceptance Criteria**
- Valid thumbnail can be attached.
- Invalid file types are rejected.
- Oversized files are rejected.
- Thumbnail reference is stored successfully.

### US-E2-04 — SuperAdmin Can Publish Program to Tenant Catalog
As a SuperAdmin, I want to publish a Program to the tenant-wide catalog so that all tenant users can discover it.

**Acceptance Criteria**
- Required fields must be present before publish.
- Published Program becomes tenant-visible.
- Publish timestamps are stored.
- Audit record is created.
- Promoted flag, if true, is stored but does not yet change base publish behavior in this Epic.

### US-E2-05 — SuperAdmin Can Edit Program Metadata
As a SuperAdmin, I want to modify Program information after creation so that data stays accurate.

**Acceptance Criteria**
- Existing Program can be updated.
- Validation is enforced.
- Updated timestamps and user identity are stored.
- Tenant and ownership integrity remain preserved.
- Promoted flag can be changed by SuperAdmin.

### US-E2-06 — Visibility Rules Are Enforced
As a product owner, I want Program visibility to obey tenant and lifecycle rules so that users only see appropriate Programs.

**Acceptance Criteria**
- Users only see Programs for their tenant.
- Draft, inactive, archived, and unpublished Programs are hidden.
- Future-dated and expired Programs behave correctly.
- Promoted flag does not bypass visibility rules in this Epic.
- Admin views can still access all Program records.

### US-E2-07 — Program Actions Are Auditable
As a compliance stakeholder, I want Program actions to be traceable so that governance is maintained.

**Acceptance Criteria**
- Create, update, publish, and status changes are logged.
- Promoted flag changes are logged.
- Audit record includes tenant, actor, Program, action, and timestamp.

### US-E2-08 — Schema Supports Future Creator-Owned Programs
As a platform architect, I want the Program model to support future Company and Professional ownership so that later workflows can be added safely.

**Acceptance Criteria**
- Ownership scope supports future actors.
- Owner entity field exists.
- Narrower catalog visibility values exist.
- Publication state supports approval workflow later.

### US-E2-09 — Schema Supports Future Promoted Program Behavior
As a platform architect, I want the Program model to support future differentiated handling for promoted Programs so that future promotion logic can be added without schema redesign.

**Acceptance Criteria**
- Program schema includes a `promoted` field.
- `promoted` is editable and auditable.
- `promoted` does not conflict with status or publication state.
- Future promoted workflow dependency is documented.

## 21. Risks and Considerations

- If ownership and publication concepts are not separated now, future creator-led publishing workflows will require redesign.
- If tenant visibility is not modeled explicitly, later access-control behavior may become inconsistent.
- If facilitator is modeled as a user reference prematurely, unnecessary coupling will be introduced.
- If promoted behavior is hardcoded now before its business definition is finalized, future changes may create rework.
- If hard delete is allowed too early, historical analytics and auditability may be compromised.
