## 1. Purpose

This document defines the business and functional requirements for Event Management and Tenant Event Publishing in StudioVerse.

The purpose of this Epic is to enable the SuperAdmin to create, manage, publish, and promote Events for a specific tenant so that those Events become visible to all Companies, Professionals, and Individuals within that tenant, and selected Events can be elevated on the tenant landing page.

This Epic establishes the canonical Event entity for StudioVerse and creates the foundation for future Event-related capabilities such as registration, reminders, ticketing, referrals, attendance, creator-led publishing workflows, and promoted landing-page placement.

This PRD is intended to provide a shared understanding across Business, Product, Engineering, QA, and Integrations teams so that all stakeholders align on scope, lifecycle behavior, data design, tenant visibility, promotion behavior, and future extensibility.

## 3. Executive Summary

Events are the community layer of StudioVerse and are intended to support webinars, workshops, masterclasses, cohort sessions, and other live or scheduled engagements across tenants.

This Epic introduces the first governed Event management capability by enabling the SuperAdmin to create Events directly into a tenant-wide catalog from the SuperAdmin page. A SuperAdmin-created Event is always created for one specific tenant and, once published and active, is visible to all Companies, Professionals, and Individuals within that tenant.

This Epic also introduces a `promoted` attribute for Events. Promotion is not the same as publication. Publication determines whether an Event is visible in the tenant catalog. Promotion determines whether an already eligible Event is elevated and shown on the tenant landing page above other Events.

In a later phase, Companies and Professionals will be able to create their own Events, but those Events will not automatically become tenant-wide. Instead, they will require publication by the SuperAdmin through a future phase-2 workflow. That workflow is out of scope for this Epic, but the Event model defined here must support it from the start so that no schema redesign is required later.

## 4. Business Context

StudioVerse is a multi-tenant platform where each tenant requires its own governed Event catalog. Events are strategically important because they create the community heartbeat of a tenant and provide a structured way to deliver webinars, workshops, masterclasses, live sessions, and future paid engagements.

Events are expected to support:
- tenant-level event discovery,
- community engagement,
- live learning and interaction,
- future reminder and notification workflows,
- future RSVP and attendance workflows,
- future event sharing and referral behavior,
- future ticketing and monetization,
- and curated promotion on tenant landing pages.

Without a stable Event model and governed creation flow, later Event features would be forced to build on unstable assumptions and would likely require rework.

---

## 5. Problem Statement

StudioVerse requires a formal Event domain model and a controlled Event creation workflow so that the SuperAdmin can create Events for a selected tenant with the correct metadata, schedule, location address/city, lifecycle state, visibility behavior, and promotion behavior.

The system must also support future ownership models where Companies and Professionals can create their own Events, but those Events must be published by the SuperAdmin in phase 2 before receiving broader tenant visibility.

Without this capability:
- there is no canonical Event structure,
- tenant-wide Event governance is missing,
- there is no clear separation between visibility and promotion,
- future reminder, RSVP, ticketing, and attendance workflows lack a stable dependency,
- and future creator-led publishing workflows would require schema migration or redesign.

## 10. Domain Definitions

### 10.1 Tenant
A tenant is the top-level Studio context under which users, content, and visibility are organized, such as Coaching Studio, Training Studio, or Recruitment Studio.

### 10.2 Event
An Event is a tenant-scoped scheduled engagement such as a webinar, workshop, masterclass, live session, or similar time-bound experience that can be listed, discovered, and attended by users within the tenant.

### 10.3 Ownership Scope
Ownership scope defines who created the Event and the default governance boundary of that Event.

Supported ownership scopes:
- `platform`
- `company`
- `professional`

This Epic implements `platform` creation only, while preparing the schema for future `company` and `professional` event creation.

### 10.4 Catalog Visibility
Catalog visibility defines who can discover the Event.

Supported visibility scopes:
- `tenant_wide`
- `company_only`
- `professional_only`

This Epic implements tenant-wide visibility for SuperAdmin-created Events and schema support for narrower future scopes.

### 10.5 Publication State
Publication state represents whether the Event is approved for broader visibility and supports future approval workflows.

Supported values:
- `draft`
- `published`
- `pending_publication_review`
- `rejected_publication`

### 10.6 Status
Status represents the operational lifecycle of the Event.

Supported values:
- `draft`
- `published`
- `inactive`
- `archived`
- `cancelled`

### 10.7 Promoted
`promoted` is an Event-level attribute that determines whether the Event should be elevated on the tenant landing page.

Promotion is distinct from publication:
- publication controls whether the Event is visible in the tenant catalog,
- promotion controls whether the Event is surfaced prominently on the landing page.

An Event must not be treated as visible merely because it is promoted. Promotion does not replace publication.

### 10.8 Event Date and Time
Event date and time define when the Event is scheduled to occur and are mandatory Event attributes in this Epic.

### 10.9 Event Type
Event type classifies the event format and is a mandatory Event attribute in this Epic.

Supported values:
- `webinar`
- `classroom`
- `classroom_session`
- `casual_meeting`

### 10.10 Location
Location is captured using two mandatory publish-ready fields in this Epic:
- `locationAddress`
- `locationCity`

The address value may represent a physical venue, an online meeting descriptor, or a hybrid label. Separating city allows filtering and discovery by city without parsing free text.

## 12. Functional Requirements

### 12.1 Event Creation
The system shall allow the SuperAdmin to create an Event for a selected tenant from the SuperAdmin page.

The Event creation flow shall support capture of:
- Event name
- Event type
- Event short description
- Event long description
- Event thumbnail or banner image
- Event date
- Event time
- Event location address
- Event location city
- Event details
- Event video or meeting URL
- Credits required, if applicable
- Status
- Publication state
- Promoted flag
- Ownership metadata

### 12.2 Tenant Association
Every Event shall belong to exactly one tenant.

### 12.3 Scheduling
Every Event shall have an Event date and time. These fields are mandatory and required for publish readiness.

### 12.4 Location
Every Event shall have location address and location city. These fields are mandatory and required for publish readiness.

### 12.5 Ownership
Events created in this Epic shall be stored with:
- `ownershipScope = platform`
- `ownerEntityId = null`

The Event schema shall also support future ownership scopes for:
- company-created Events
- professional-created Events

### 12.6 Catalog Visibility
Events created by SuperAdmin shall use tenant-wide visibility when published.

Tenant-wide visibility means the Event is discoverable by:
- all Companies in the tenant,
- all Professionals in the tenant,
- all Individuals in the tenant.

### 12.7 Event Lifecycle
The system shall support the following status values:
- `draft`
- `published`
- `inactive`
- `archived`
- `cancelled`

The system shall support the following publication states:
- `draft`
- `published`
- `pending_publication_review`
- `rejected_publication`

### 12.8 Promotion
The system shall support a boolean `promoted` attribute on the Event entity.

In this Epic:
- `promoted` shall default to `false`,
- `promoted` may be set only by SuperAdmin,
- `promoted = true` shall make the Event eligible for elevated landing-page placement,
- `promoted = true` shall not by itself make the Event visible in the tenant catalog,
- an Event must satisfy publication and visibility rules before it can be surfaced on the landing page as a promoted Event.

### 12.9 Event Editing
The system shall allow the SuperAdmin to edit existing Event metadata after creation, including promoted status.

### 12.10 Auditability
The system shall create audit records for key Event actions including:
- create
- update
- publish
- cancel
- archive
- status change
- promoted flag change

## 14. Business Rules

- Only SuperAdmin can create Events in this Epic.
- Every Event must belong to one and only one tenant.
- Event name is required.
- Event type is required.
- Event short description is required.
- Event long description is required.
- Event date is required.
- Event time is required.
- Event location address is required.
- Event location city is required.
- Thumbnail is recommended for draft and required for publish readiness if visual consistency is required across the Event catalog.
- Video or meeting URL is optional.
- Credits required is optional in this Epic unless business decides Events follow a credit-gated model.
- `promoted` is optional and defaults to `false`.
- Setting `status = published` alone is not sufficient for tenant visibility; publication state and visibility rules must also be satisfied.
- Setting `promoted = true` does not by itself publish the Event.
- Setting `promoted = true` does not override tenant, status, or publication rules.
- Only published and eligible Events can be surfaced as promoted items on the landing page.
- Cancelled Events remain in the system for history and audit purposes.
- Events must not be hard deleted in MVP.
- Published tenant-wide Events are visible only within the tenant they belong to.

---

## 15. Visibility Rules

An Event shall be visible in the active tenant Event catalog only when all of the following are true:

- The Event belongs to the same tenant as the user.
- `status = published`
- `publicationState = published`
- The Event has not been archived.
- The Event has not been cancelled.

An Event shall not be visible in the active tenant Event catalog if any of the following are true:

- `status = draft`
- `status = inactive`
- `status = archived`
- `status = cancelled`
- `publicationState != published`

Administrative management views may display Events regardless of end-user visibility state.

For future Company-created and Professional-created Events, the default narrower visibility and later publication expansion will be defined in phase 2, but the schema must support those rules now.

---

## 16. Promotion Rules

An Event shall be eligible for promoted landing-page placement only when all of the following are true:

- `promoted = true`
- the Event belongs to the active tenant
- `status = published`
- `publicationState = published`
- the Event is not archived
- the Event is not cancelled

Promoted Events shall be displayed above non-promoted Events on the tenant landing page.

If multiple Events are promoted, the display order among promoted Events may initially follow a simple fallback such as nearest upcoming event date or most recently updated, until a dedicated promotion-ordering rule is defined in a later Epic.

If an Event becomes inactive, archived, cancelled, or unpublished, it must no longer appear as a promoted Event even if `promoted = true` remains stored on the record.

Promotion affects landing-page prominence only. It does not replace standard catalog visibility logic.

This Epic must support future extension to the following:

### 17.1 Company-Created Events
Future Company-created Events must be able to use the same Event schema while storing:
- `ownershipScope = company`

### 17.2 Professional-Created Events
Future Professional-created Events must be able to use the same Event schema while storing:
- `ownershipScope = professional`
- `ownerEntityId = professionalId`
- default `catalogVisibility = professional_only`

### 17.3 Event Publication Request Workflow
A future Epic will introduce a publication request workflow whereby Company-created and Professional-created Events can be submitted for SuperAdmin review and approval before becoming tenant-wide.

### 17.4 Event Engagement Features
Future epics may introduce:
- RSVP,
- reminders,
- attendee communication,
- sharing/referral,
- attendance tracking,
- and paid ticketing.

### 17.5 Promotion Enhancements
Future epics may introduce:
- explicit promotion ordering,
- promotion scheduling windows,
- limited featured slots,
- promoted analytics,
- and configurable landing-page merchandising rules.

The Event schema defined here must support those later capabilities without redesign.

## 18. User Stories

### US-E3-01 — Define Event Domain Model
**As a** developer,  
**I want** a canonical tenant-aware Event model,  
**so that** all current and future Event features use one stable schema.

**Acceptance Criteria**
- Event schema supports tenant, ownership, visibility, publication, scheduling, location address/city, lifecycle, and promoted flag.
- Event schema supports tenant, ownership, visibility, publication, event type, scheduling, location address/city, lifecycle, and promoted flag.
- Schema supports platform, company, and professional ownership.
- Event includes required date, time, location address, and location city fields.
- Validation requirements are documented.
- Schema supports future publication review workflows.
- Schema supports future landing-page promotion workflows.

---

### US-E3-02 — SuperAdmin Can Create a Draft Event
**As a** SuperAdmin,  
**I want** to create an Event in draft mode for a selected tenant,  
**so that** I can prepare Event details before publishing it.

**Acceptance Criteria**
- Tenant is selected during create flow.
- Event draft can be saved successfully.
- Draft Event is stored as platform-owned.
- Draft Event is not visible in the active tenant Event catalog.
- Promoted flag can be set or left unset during creation, but does not affect visibility until publish rules are satisfied.

---

### US-E3-03 — SuperAdmin Can Add Event Type, Schedule, and Location Details
**As a** SuperAdmin,  
**I want** to define event type, date, time, location address, and city for an Event,  
**so that** users know when and where the Event will take place.

**Acceptance Criteria**
- Event type can be selected.
- Event date can be entered.
- Event time can be entered.
- Event location address can be entered.
- Event location city can be entered.
- Invalid or incomplete schedule input is rejected for publish readiness.
- Canonical Event scheduling data is stored successfully.

---

### US-E3-04 — SuperAdmin Can Publish Event to Tenant Catalog
**As a** SuperAdmin,  
**I want** to publish an Event to the tenant-wide Event catalog,  
**so that** all tenant users can discover it.

**Acceptance Criteria**
- Required fields must be present before publish.
- Published Event becomes tenant-visible.
- Publish timestamp is stored.
- Audit record is created.
- Promoted flag, if true, does not bypass publish validation.

---

### US-E3-05 — SuperAdmin Can Promote an Event
**As a** SuperAdmin,  
**I want** to promote an Event,  
**so that** it appears on the tenant landing page above other Events.

**Acceptance Criteria**
- SuperAdmin can set `promoted = true` or `promoted = false`.
- Only eligible published Events appear as promoted on the landing page.
- Promoted Events appear above non-promoted Events on the landing page.
- Unpublished, archived, inactive, or cancelled Events do not appear as promoted even if the flag remains true.
- Promotion changes are auditable.

---

### US-E3-06 — SuperAdmin Can Edit Event Metadata
**As a** SuperAdmin,  
**I want** to modify Event information after creation,  
**so that** Event information stays accurate.

**Acceptance Criteria**
- Existing Event can be updated.
- Validation is enforced.
- Updated timestamps and user identity are stored.
- Tenant and ownership integrity remain preserved.
- Promoted flag can be changed by SuperAdmin.

---

### US-E3-07 — Event Visibility Rules Are Enforced
**As a** Product Owner,  
**I want** Event visibility to obey tenant and lifecycle rules,  
**so that** users only see appropriate Events.

**Acceptance Criteria**
- Users only see Events for their own tenant.
- Draft, inactive, archived, cancelled, and unpublished Events are hidden from the active tenant Event catalog.
- Admin views can still access all Event records.

---

### US-E3-08 — Event Promotion Rules Are Enforced
**As a** Product Owner,  
**I want** promoted Event surfacing to obey publish and lifecycle rules,  
**so that** only valid Events are elevated on the landing page.

**Acceptance Criteria**
- Promoted Events must also satisfy publish visibility requirements.
- Promoted Events appear above non-promoted Events on the landing page.
- Invalid promoted Events are excluded from promoted display.
- Promotion does not make an Event visible if it is otherwise unpublished or inactive.

---

### US-E3-09 — Event Actions Are Auditable
**As a** compliance stakeholder,  
**I want** Event actions to be traceable,  
**so that** governance is maintained.

**Acceptance Criteria**
- Create, update, publish, cancel, archive, status changes, and promoted flag changes are logged.
- Audit records include tenant, actor, Event, action, and timestamp.

---

### US-E3-10 — Schema Supports Future Creator-Owned Events
**As a** platform architect,  
**I want** the Event model to support future Company and Professional ownership,  
**so that** later workflows can be added safely.

**Acceptance Criteria**
- Ownership scope supports future actors.
- Owner entity field exists.
- Narrower catalog visibility values exist.
- Publication state supports approval workflow later.

## 22. Risks and Considerations

- If ownership and publication concepts are not separated now, future creator-led Event publishing workflows will require redesign.
- If publish and promote concepts are conflated, landing-page behavior and catalog behavior will become inconsistent.
- If scheduling is modeled loosely, later reminder, calendar, and attendance integrations will become harder.
- If location address and city are not captured as first-class fields now, later event discovery, city filtering, and communication flows may become inconsistent.
- If hard delete is allowed too early, historical analytics and auditability may be compromised.
