## 1. Purpose

This document defines the business and functional requirements for the Tools module in StudioVerse, with primary focus on the Assessment Engine.

The purpose of this Epic is to enable StudioVerse to create, manage, publish, assign, deliver, score, and analyze structured Assessments across Coaching Studio, Training Studio, and Recruitment Studio using a reusable and scalable engine instead of one-off custom implementations.

This Epic establishes the canonical Assessment model for StudioVerse and provides the foundation for AI-driven narrative reporting, embeddable assessment delivery, future question-generation workflows, longitudinal data capture, and gamified assessment experiences.

This PRD is intended to serve as the functional source of truth for product, engineering, QA, design, and implementation teams, and may also be used as an input specification for AI coding tools such as GitHub Copilot.

## 3. Executive Summary

The Tools module is one of the most strategically differentiated parts of StudioVerse because it does more than display content. It captures structured participant response data, interprets that data, and turns it into meaningful professional insight through AI-generated analysis and reports.

Within this module, the official term will be **Assessment** rather than Test.

An Assessment is a structured, purpose-driven instrument used to evaluate a participant’s awareness, capabilities, style, readiness, preferences, strengths, or developmental profile in a specific context.

This Epic introduces a reusable Assessment Engine in which multiple Assessments can be created from common patterns. Although StudioVerse may host 12 to 20 Assessments in the initial wave, those Assessments will be delivered through a small number of standard render styles rather than one custom page per Assessment.

Each Assessment will have its own identity, context, participant benefit, AI analysis prompt, question bank, and delivery configuration. At runtime, the platform will fetch the appropriate number of questions for that Assessment, select the correct renderer based on metadata, capture participant answers, and send the structured output to an AI model such as Groq or Gemini for narrative report generation.

---

## 4. Business Context

Across coaching, training, and recruitment use cases, Assessments are not simply quizzes. They are structured mechanisms for understanding a participant more deeply, identifying developmental patterns, and guiding programs and interventions more effectively.

In Coaching Studio in particular, the foundational direction for this capability comes from leadership-oriented Assessments previously used in thecoachdinesh.com to help evaluate skills, capabilities, blind spots, and leadership potential.

StudioVerse now requires a scalable platform-native version of that capability. The platform must support many Assessments, but it should not implement each one as a separate custom product artifact. Instead, Assessments must run on a shared engine so that authoring, delivery, scoring, AI reporting, and future analytics can scale efficiently.

---

## 5. Problem Statement

StudioVerse currently needs a formal Assessment domain model and a controlled Assessment Engine architecture so that the platform can define Assessment metadata, store question banks, render different interaction styles, capture participant answers, and generate AI-driven reports.

The platform must also support different styles of Assessment delivery. Some Assessments will behave like standard single-choice quizzes, while others will include instant feedback, image-based questions, selection-transfer interactions, and future gamified experiences.

Without this capability:
- there is no canonical Assessment structure,
- Assessment experiences would be hardcoded one by one,
- AI reporting would lack Assessment-specific prompts and context,
- question storage and reuse would be inconsistent,
- and the platform would not have a scalable foundation for future Assessment growth.

---

## 6. Objectives

The objectives of this Epic are:

- Establish Assessment as the canonical terminology and entity in the Tools module.
- Enable SuperAdmin to create and manage Assessments for a tenant.
- Support Assessment metadata such as short description, long description, context, and participant benefit.
- Support multiple reusable Assessment render styles.
- Store an Assessment-specific AI analysis prompt for result interpretation.
- Optionally store an Assessment-specific AI question-generation prompt for authoring support.
- Store question banks against an Assessment rather than hardcoding questions into page components.
- Allow the platform to fetch only the configured number of questions for a given Assessment attempt.
- Support AI-generated narrative reports through Groq, Gemini, or equivalent backend AI services.
- Integrate Assessments with assignment workflows, report delivery, and future widget-based usage.
- Future-proof the module for richer gamification and creator-led Assessment authoring.

---

## 7. Non-Objectives

The following are explicitly out of scope for this Epic:

- Creating 12 to 20 custom-coded pages for each individual Assessment
- Fully dynamic live question generation during participant runtime
- Advanced psychometric calibration and norming
- Adaptive testing that changes difficulty based on prior answers
- Assessment marketplace pricing and monetization rules
- Enterprise benchmarking dashboards
- Full multilingual localization in the first release
- Real-time multiplayer assessment experiences
- Fully autonomous, unreviewed AI-generated question delivery at runtime

This Epic includes the following capabilities:

- SuperAdmin creates and manages Assessment definitions
- Assessment metadata is stored in the datastore
- Questions are associated with an Assessment through Assessment ID
- Assessment-specific AI analysis prompts are stored
- Optional Assessment-specific AI question-generation prompts are stored
- Runtime fetches the configured number of questions for an Assessment attempt
- Assessment delivery is driven by metadata and render style

## 10. Domain Definitions

### 10.1 Tool
A Tool is a functional interactive resource within StudioVerse that allows a user to provide input and receive a structured result, outcome, or insight.

### 10.2 Assessment
An Assessment is a type of Tool that presents a structured set of questions or interactive tasks to a participant in order to evaluate awareness, capability, preferences, behavior, readiness, or other development-related dimensions.

### 10.3 Assessment Question Bank
A Question Bank is the collection of approved questions associated with a specific Assessment and stored against that Assessment for runtime selection.

### 10.4 Assessment Attempt
An Assessment Attempt is a single participant interaction session in which the platform fetches questions, records answers, calculates raw outputs, and generates or requests a final report.

### 10.5 AI Analysis Prompt
An AI Analysis Prompt is the Assessment-specific instruction template used by the backend AI service to interpret submitted answers and generate a meaningful narrative report.

### 10.6 AI Question Generation Prompt
An AI Question Generation Prompt is an optional Assessment-specific instruction template used during admin authoring to generate draft question suggestions for review and storage.

### 10.7 Render Style
Render Style defines how the Assessment is visually presented and how the participant interacts with the Assessment on the screen.

## 11. User Roles

### 11.1 SuperAdmin

### 11.2 Professional

## 12. Functional Requirements

### 12.1 Assessment Creation
The system shall allow the SuperAdmin to create an Assessment for a selected tenant.

The Assessment creation flow shall support capture of:
- Assessment name
- Assessment short description
- Assessment long description
- Assessment context
- Assessment benefit
- Assessment type
- Render style
- Number of questions in bank
- Number of questions to present per attempt
- AI analysis prompt
- Optional AI question-generation prompt
- Status
- Publication state
- Ownership metadata

### 12.2 Assessment Terminology
The platform shall use the term **Assessment** in product language, documentation, admin forms, participant-facing screens, and system metadata rather than the term Test.

### 12.3 Assessment Context Fields
Each Assessment shall support richer descriptive metadata so that both the participant experience and the AI interpretation remain grounded in context and purpose rather than only raw scores.

At minimum, the system shall store:
- short description,
- long description,
- Assessment context,
- and participant benefit.

### 12.4 Question Bank Association
Each Assessment shall support an associated Question Bank stored in the datastore and linked by `assessmentId` rather than hardcoded into page components.

### 12.5 Question Count Configuration
The system shall allow each Assessment to define:
- total question count available in the bank,
- and the number of questions to be fetched and presented for a participant attempt.

The system shall not require all stored questions to be displayed in every attempt.

### 12.6 Runtime Delivery
At runtime, the system shall:
- load Assessment metadata,
- fetch the configured set of questions,
- choose the correct render style,
- display the questions using the appropriate reusable renderer,
- collect participant responses,
- and submit the result payload for scoring and AI interpretation.

### 12.7 AI Analysis Prompt
Each Assessment shall store an Assessment-specific AI analysis prompt used after submission to interpret the participant’s responses and generate the report.

### 12.8 AI Question Generation Prompt
The system should optionally support a separate AI prompt for generating draft questions during Assessment authoring so that question creation and answer interpretation remain separate concerns.

### 12.9 Reusable Renderers
The frontend shall use a reusable renderer model rather than a separate custom page for every Assessment.

This means the platform shall maintain a small number of reusable render-style pages or components and use metadata to decide which one to render for a given Assessment.

### 12.10 Results and Reports
After an Assessment Attempt is completed, the system shall:
- store the participant’s submitted answers,
- produce raw result data,
- send structured Assessment context plus answers to the configured AI provider,
- receive AI analysis,
- create a report record,
- and make the report available to the appropriate users.

## 13. Supported Render Styles

The first release of the Assessment Engine shall support the following render styles.

### 13.1 Single-Choice Quiz
A participant sees a question and selects one answer from multiple radio-button options.

### 13.2 Instant-Feedback Multi-Choice
A participant sees multiple options and receives immediate green or red feedback when selecting an answer, creating a more interactive and game-like experience.

### 13.3 Select-and-Move / Select-All-That-Apply
A participant sees a list of items on one side and selects items which move to another area, visually representing chosen responses.

### 13.4 Image-Based Single-Choice
A participant sees an image, supporting descriptive text or scenario text, and then answers a single-choice question based on that visual context.

### 13.5 Gamified Style
The platform should support a fifth, more gamified style. The preferred candidate is a drag-and-drop matching or ordering interaction because it is interactive, intuitive, and still manageable in an MVP architecture.

### 13.6 Future Render Styles
The engine shall be designed so that future render styles can be added without redesigning the whole Assessment model.

## 15. Business Rules

- The platform shall use **Assessment** as the official term for this capability.
- An Assessment must belong to one tenant.
- Questions must be associated with an Assessment by `assessmentId`.
- Questions must not be hardcoded into 12 to 20 individual frontend pages.
- The number of questions shown in an attempt may be fewer than the number stored in the Question Bank.
- AI analysis prompts must be stored per Assessment, not as one global prompt for all Assessments.
- AI question-generation prompts and AI analysis prompts should be treated as separate concerns.
- Runtime Assessment behavior must be chosen from metadata rather than custom code per Assessment.
- External AI model calls must happen from backend services only, not from the frontend.
- Assessment completion should support assignment completion workflows when the Assessment was assigned through the platform.

---

## 16. Architectural Recommendation

The platform should **not** create separate `page.tsx` files for every Assessment. That approach may seem fast initially but will create duplication, inconsistency, and high maintenance overhead as the number of Assessments grows.

Instead, the recommended architecture is:
- one generic Assessment runtime route,
- a small set of reusable renderer components or pages,
- metadata-driven renderer selection,
- separate Assessment and Question storage,
- and backend-driven scoring and AI analysis.

This approach is more scalable, more consistent, easier to test, and better aligned with StudioVerse’s configuration-driven product philosophy.

---

## 17. AI Analysis Model

Each Assessment shall carry a dedicated AI analysis prompt because the meaning of the participant’s answers depends on the specific purpose of the Assessment.

For example, an Assessment focused on self-awareness requires a different interpretive lens than an Assessment focused on decision-making, leadership style, blind spots, or developmental readiness.

Therefore, the system must send not only answers and scores, but also the Assessment context, benefit, and prompt instructions when generating the final report.

The platform should also support provider abstraction so that report generation may be performed by Groq, Gemini, or equivalent approved AI services without changing the Assessment model.

**Acceptance Criteria**

- Correct renderer is selected.

**As a** Product Owner,  
**I want** each Assessment to control how many questions appear per attempt,  
**so that** the platform can support both fixed and subset-driven delivery.

- Assessment stores `questionsPerAttempt`.
- Runtime fetches only the configured number.
- The Question Bank can contain more questions than the attempt displays.

**As a** Professional,  
**I want** submitted Assessment answers to be analyzed in the right context,  
**so that** the resulting report is meaningful and useful.

- Assessment-specific AI prompt is used.
- Structured answers are sent to backend AI services.
- AI response is stored with the report record.

---

### US-E4-07 — Assessment Completion Updates Assignment
**As a** Professional,  
**I want** assigned Assessments to update completion status when finished,  
**so that** assignment tracking remains accurate.

**Acceptance Criteria**
- If an Assessment is assignment-linked, completion updates assignment state.
- Report generation can happen after submission.
- Appropriate users can access the output.

## 22. Risks and Considerations

- If each Assessment is custom-built as a separate page, maintenance cost will rise quickly and the engine will not scale.
- If AI prompts are global rather than Assessment-specific, report quality will decline because the analysis will ignore Assessment context.
- If question-generation prompts and analysis prompts are not separated, the platform will mix authoring concerns with interpretation concerns.
- If questions are generated live at runtime without prior review, content quality and reliability may become inconsistent.
- If render styles are not modeled explicitly, frontend complexity will increase and reuse will decrease.
