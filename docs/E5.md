## 1. Purpose

This document defines the business and functional requirements for the **Manage Wallet** capability in StudioVerse.

The purpose of this Epic is to allow the **SuperAdmin** to issue platform coins to tenant entities and end users, and allow recipients to see their available balance within **My Wallet** after login.

This Epic also establishes the administrative controls, wallet visibility model, transaction tracking, dashboard summary behavior, and future-ready ledger structure required for controlled coin-based access and usage across the StudioVerse platform.

## 3. Executive Summary

StudioVerse requires a platform-native mechanism to issue and track **coins** that can be assigned by the SuperAdmin to Companies, Coaches / Professionals, or Individuals.

Once coins are issued, the logged-in recipient must be able to see their balance in **My Wallet**, creating a direct and transparent connection between administrative allocation and user-level visibility.

The SuperAdmin must also have a dedicated management entry point called **Manage Wallet**, where wallets can be reviewed and coins can be assigned after selecting the tenant and the target user type.

In addition, the SuperAdmin dashboard must contain a summary tile that shows overall platform utilization in a form such as **10/20**, where:

- **10** = total utilized coins
- **20** = total issued coins across all tenants

Selecting that tile must take the SuperAdmin directly to the Manage Wallet page.

---

## 4. Business Context

As StudioVerse evolves into a structured multi-tenant platform with centrally governed experiences, it requires an internal allocation model that allows the platform owner to control usage and entitlement in a measurable way.

Coins provide that mechanism. They can represent access value, usage credits, or controlled platform entitlement units that the SuperAdmin can allocate across the ecosystem.

This model is especially valuable in a multi-tenant setup because allocation may need to happen at more than one level:

- to a Company / Tenant,
- to a Coach / Professional,
- or directly to an Individual user.

---

## 5. Problem Statement

StudioVerse currently needs a structured Wallet capability that allows coins to be issued, surfaced to users, and tracked consistently across the platform.

Without this capability:

- there is no canonical place to assign coins,
- users cannot clearly see what has been allocated to them,
- administrators cannot track how many coins have been issued or used,
- and the platform lacks the ledger foundation needed for future controlled consumption use cases.

The platform also requires a clear separation between:

- **coin issuance**,
- **coin visibility**, and
- **coin utilization**,

so that the system can accurately report assigned balance versus actual usage over time.

---

## 6. Objectives

The objectives of this Epic are:

- Enable SuperAdmin to assign coins to a Company, Coach / Professional, or Individual.
- Allow logged-in users to see their coin balance in **My Wallet**.
- Add a dedicated **Manage Wallet** menu item to the SuperAdmin navigation.
- Allow the SuperAdmin to choose a tenant before assigning coins.
- Allow the SuperAdmin to choose a target user type before assigning coins.
- Record every allocation in a transaction-safe ledger structure.
- Track total issued coins across all tenants.
- Track utilized coins separately from issued coins.
- Show a dashboard tile summarizing utilized vs issued totals in a format such as 10/20.
- Allow the dashboard tile to deep-link to the Manage Wallet page.
- Future-proof the system for controlled debit / consumption scenarios.

## 10. Domain Definitions

### 10.1 Coin
A Coin is an internal StudioVerse value unit issued by the platform for controlled entitlement, balance tracking, and future usage consumption.

### 10.2 Wallet
A Wallet is the system-managed balance container through which a logged-in user or entity can view issued, used, and available coin values.

### 10.3 Coin Issuance
Coin Issuance is the action by which the SuperAdmin allocates a specified number of coins to a valid target entity.

### 10.4 Wallet Transaction
A Wallet Transaction is a ledger record representing a balance-affecting event such as issue, adjustment, debit, reversal, or expiry.

### 10.5 Utilized Coins
Utilized Coins are coins that have already been consumed by a valid platform action.

### 10.6 Unused Coins
Unused Coins are coins that have been issued but not yet utilized.

### 10.7 Target User Type
The Target User Type defines who receives the coin allocation:
- Company / Tenant
- Coach / Professional
- Individual

---

## 11. User Roles

### 11.1 SuperAdmin
The SuperAdmin can issue coins, review allocation data, and track issued versus utilized totals across the platform.

### 11.2 Company / Tenant
A Company may receive coins through the SuperAdmin allocation flow and may later use those coins for platform-controlled activity.

### 11.3 Coach / Professional
A Coach / Professional may receive coins directly through the SuperAdmin flow and must be able to view them in My Wallet once logged in.

### 11.4 Individual
An Individual may receive coins directly and must be able to see the assigned balance in My Wallet after login.

## 12. Functional Requirements

### 12.1 SuperAdmin Navigation
The system shall add a new menu item for SuperAdmin called **Manage Wallet**.

### 12.2 Dashboard Tile
The SuperAdmin dashboard shall contain a tile showing:
- total utilized coins,
- total issued coins,
- and a compact ratio-style summary such as **10/20**.

The tile shall be clickable and shall navigate the user to the **Manage Wallet** page.

### 12.3 Manage Wallet Page
The system shall provide a SuperAdmin-only page for wallet management.

The page shall support:
- existing wallet listing,
- user-type radio filtering,
- tenant selection,
- target user type selection,
- target entity lookup / selection,
- coin quantity entry,
- issuance action,
- wallet snapshot for selected user,
- and role-aware wallet operations (Company coin request approvals, Professional coin requests in tenant wallet routes).

### 12.4 Tenant Selection
Before assigning coins, the SuperAdmin shall be able to select the target tenant.

### 12.5 Target Type Selection
After selecting the tenant, the SuperAdmin shall be able to choose the type of recipient:
- Company / Tenant
- Coach / Professional
- Individual

### 12.6 Target Entity Selection
Once the target type is selected, the system shall present valid entities from that tenant for selection, except in the case of a Company-level allocation where the tenant itself may be the target entity.

### 12.7 Coin Assignment
The SuperAdmin shall be able to assign a positive integer number of coins to the selected target entity.

### 12.8 Wallet Visibility
When coins are assigned to a user-level target, the logged-in recipient shall be able to see those coins in **Manage Wallet / My Wallet**.

### 12.9 Wallet Summary
Manage Wallet / My Wallet shall show at minimum:
- total issued coins,
- total utilized coins,
- available / unused coins,
- and wallet transaction history including debit reasons where applicable.

### 12.10 Issuance Ledger
Every coin assignment shall create a ledger transaction record with metadata such as:
- issuer,
- tenant,
- target type,
- target entity,
- quantity,
- timestamp,
- and reason / note where applicable.

### 12.11 Utilization Tracking
The system shall separately track utilization events so that issued and used totals are not conflated.

### 12.12 Balance Calculation
Available coin balance shall be calculated as:

**Available Coins = Total Issued Coins - Total Utilized Coins - Other Valid Debits + Valid Credits**

For MVP, the primary calculation may be simplified to:

**Available Coins = Total Issued Coins - Total Utilized Coins**

### 12.13 Auditability
The system shall preserve auditable wallet transactions and shall not depend only on a mutable current balance field.

### 12.14 Role-Based Access
Only the SuperAdmin shall be able to issue coins in this version of the Epic.

4. Review issued and utilized values.
5. Review recent wallet history where applicable.

## 15. Business Rules

- Only SuperAdmin may assign coins in this release.
- Coin assignment must be tenant-aware.
- A target recipient must belong to the selected tenant.
- Coin quantity must be a positive whole number.
- Coin assignment must create a ledger transaction.
- My Wallet must reflect valid issued balance for the logged-in owner.
- Utilized coins must be tracked separately from issued coins.
- Unused coins must equal issued coins minus utilized coins, unless additional debit rules are later introduced.
- Dashboard summary must aggregate totals across all tenants.
- Clicking the wallet summary tile must open the Manage Wallet page.
- The system must preserve auditability and should not rely only on overwriting one balance field.

---

## 16. Architectural Recommendation

The wallet system should be implemented using a **ledger-first model** rather than a balance-only model.

That means:
- every allocation or usage event creates a transaction record,
- balances may be stored as denormalized summary fields for performance,
- but the source of truth remains the transaction history.

This is recommended because it improves:
- auditability,
- debugging,
- reconciliation,
- future reporting,
- and extensibility for coin-based platform usage.

The platform should maintain:
- a wallet entity for current summarized balance,
- and a wallet transaction collection for immutable balance events.

---

## 17. Dashboard and Reporting Model

The SuperAdmin dashboard must present a compact but meaningful summary of wallet state through a tile such as:

**Utilized / Issued = 10 / 20**

Interpretation:
- **10** = total utilized coins
- **20** = total issued coins across all tenants

The system should also be capable of deriving:
- unused coins,
- utilization rate,
- and allocation coverage by tenant,
even if only the utilized/issued tile is shown in the first release.

### US-E5-10 — New Registrations Receive Tenant-Configured Bonus Coins (Completed)
**As a** newly registered user,  
**I want** my wallet to be initialized with my tenant's registration free coins,  
**so that** I can start using wallet-based flows immediately after onboarding.

**Acceptance Criteria**
- During registration, wallet creation reads `tenants/{tenantId}.walletConfig.registrationFreeCoins` (fallback 10).
- Wallet is created with `totalIssuedCoins` and `availableCoins` set to the resolved registration bonus.
- A `walletTransactions` credit entry is created with reason `Registration bonus`.
- Behavior is tenant-aware and applies across current registration entry points.

---

### US-E5-08 — Professional Can Request Coins from Associated Company (Completed)
**As a** Professional,  
**I want** to submit a coin request to my associated Company,  
**so that** I can request additional coins for platform usage without requiring SuperAdmin coordination.

**Acceptance Criteria**
- Request Coins page is accessible at `/<tenant>/request-coins`.
- Professional can enter a quantity and submit a request.
- Request is stored in Firestore with status "pending" and includes Professional ID and associated Company ID.
- Firestore rules allow Professional create if requesterProfessionalId matches auth.uid and status is "pending".
- Request is routed to the correct Company based on Professional.associatedCompanyId from profile context.

---

### US-E5-09 — Company Can Approve or Deny Coin Requests with Ruleset Enforcement (Completed)
**As a** Company user,  
**I want** to view, approve, and deny pending coin requests from associated Professionals,  
**so that** I can govern coin distribution to team members responsibly.

**Acceptance Criteria**
- Manage Wallet page shows pending coin request count as a live badge.
- Company users can click to open coin requests modal.
- Modal displays all pending requests for the Company.
- Approve action transitions request status to "approved" and adds coins to Professional's wallet via ledger transaction.
- Deny action transitions request status to "denied" with optional reason field.
- Firestore rules restrict approve/deny to Company users only (isCompanyUser() constraint).
- Badge count refreshes in real-time after action is taken.
- Delete operations are permanently disabled (soft-delete pattern only).

## 19. QA Considerations

QA should validate the following scenarios:

- SuperAdmin can see Manage Wallet navigation.
- Dashboard tile displays correct utilized and issued totals.
- Clicking the tile opens Manage Wallet.
- Tenant selection filters available target entities correctly.
- Company-level, Coach-level, and Individual-level assignment flows work correctly.
- Invalid tenant-target combinations are blocked.
- Coin quantity validation prevents zero, negative, or invalid values.
- Issuance creates a ledger transaction.
- Wallet summaries update after issuance.
- My Wallet shows correct values for the logged-in owner.
- Utilized and unused calculations are correct.
- Cross-tenant aggregation works correctly for dashboard totals.
- Unauthorized roles cannot issue coins.

**Registration Bonus (US-E5-10) Testing:**
- Set tenant `walletConfig.registrationFreeCoins` to a non-default value (for example 10).
- Register a brand-new eligible user and verify wallet doc is created on onboarding.
- Verify `totalIssuedCoins` equals tenant registration free coins.
- Verify `availableCoins` equals tenant registration free coins.
- Verify wallet transaction ledger contains credit reason `Registration bonus`.
- Verify no duplicate registration bonus is applied on subsequent logins.

**Coin Request Lifecycle (US-E5-08/09) Testing:**
- Professional can access `/<tenant>/request-coins` page.
- Professional can submit a coin request with valid quantity.
- Firestore rules enforce: requests can only be created by the requesting Professional (auth.uid matches requesterProfessionalId).
- Firestore rules enforce: requests with invalid status or zero amount are rejected.
- Professional requests are routed to the correct Company based on profile association.
- Company users see pending coin request count on Manage Wallet page badge.
- Badge count is accurate and updates after approval/denial.
- Company users can open modal and view all pending requests for their company.
- Company users can approve a request (status transitions to "approved", coins added to Professional's wallet).
- Company users can deny a request (status transitions to "denied").
- Firestore rules restrict approve/deny operations to Company users only.
- Unauthorized roles (Individual, Professional without company role) cannot approve/deny requests.
- Hard delete is disabled for coin requests (only soft-delete via status updates permitted).
- Coin request transactions appear in Professional wallet history after approval.
- Coin request transactions appear in Company audit/transaction history.

## 22. Risks and Considerations

- If the system stores only a mutable balance and no transaction ledger, auditability will be weak.
- If tenant scoping is not enforced, coins may be assigned to the wrong entity.
- If utilized and issued values are not separated, dashboard reporting will be misleading.
- If My Wallet does not clearly distinguish available vs utilized, users may misunderstand their balance.
- If company-, coach-, and individual-level ownership rules are not modeled carefully, later consumption logic may become difficult.
