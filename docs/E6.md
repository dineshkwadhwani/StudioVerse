## 1. Purpose

This document defines the business and functional requirements for the **Manage Profile** capability in StudioVerse.

The purpose of this Epic is to create a role-aware profile system that captures the minimum required information needed for users to participate meaningfully in the platform while allowing a stepped approach to profile enrichment over time.

This Epic also establishes the profile-completion gating behavior required before an Individual can be assigned an Assessment, Program, or Event, and lays the foundation for richer assessment quality, recruitment usage, search, matching, and future public profile scenarios such as **Search Coach**.

## 3. Executive Summary

StudioVerse needs a structured Profile capability that supports the practical reality that users cannot be expected to complete a very large form all at once.

Therefore, the platform will follow a **stepped profile completion approach**. On first access to the dashboard, users who have not completed the mandatory part of their profile will be shown a top-level prompt asking them to complete their profile.

Selecting that prompt will take the user to an **Update Profile** page that contains a full profile layout with multiple sections. The mandatory section must be completed before an Individual becomes eligible to receive platform assignments such as Assessments, Programs, or Events.

All remaining sections may remain optional in the early stage, but the product will explicitly guide the user by stating that richer profile information can lead to **better assessments** and stronger platform outcomes.

The Profile object will support multiple role types:
- **Individual**
- **Professional**
- **Company**

Professionals will have additional professional-facing fields, and in a future Epic such as **Search Coach**, that information can power a world-visible public coach profile.

Companies will also have their own basic profile structure including company identity and location data.

---

## 4. Business Context

StudioVerse is a multi-tenant platform spanning coaching, training, and recruitment-oriented scenarios, which makes Profile more than a simple personal information form.

Profile data can support:
- onboarding readiness,
- assignment eligibility,
- better contextual assessments,
- professional and recruitment referrals,
- future search and discovery,
- tenant-aware data completeness,
- and better decision-making across the platform.

Because the platform includes Individuals, Professionals, and Company entities, a one-size-fits-all profile form would be too rigid and too heavy. A role-aware and phased model is therefore required.

---

## 5. Problem Statement

StudioVerse currently requires a formal Profile capability that:
- captures the minimum mandatory data needed for platform participation,
- supports role-aware extensions,
- and avoids forcing users to fill everything in one go.

The platform also needs a clear mechanism to ensure that assignment-related experiences do not begin before the Individual has provided the minimum necessary profile information.

Without this capability:
- users may enter core journeys with incomplete identity data,
- assessments may lack useful context,
- assignments may fail operationally,
- profile quality will vary significantly,
- and the platform will not have a reliable foundation for future search, recruitment, and public profile use cases.

---

## 6. Objectives

The objectives of this Epic are:

- Establish Profile as a canonical role-aware platform object.
- Support profile creation and update for Individuals, Professionals, and Companies.
- Define mandatory versus optional fields clearly.
- Support a stepped profile completion experience instead of forcing full completion upfront.
- Show a dashboard prompt requesting profile completion when mandatory data is missing.
- Route users from the dashboard prompt to an **Update Profile** page.
- Gate assignment eligibility for Individuals until mandatory profile fields are complete.
- Support optional profile enrichment through structured sections.
- Encourage optional enrichment by communicating that richer profiles lead to better assessments.
- Support optional profile photo upload.
- Lay groundwork for future public Professional profile visibility in Search Coach.

## 10. Domain Definitions

### 10.1 Profile
A Profile is the structured identity and contextual information object associated with a platform actor such as an Individual, Professional, or Company.

### 10.2 Mandatory Profile Fields
Mandatory Profile Fields are the minimum set of fields that must be completed in order for the user to be considered profile-complete for core participation.

### 10.3 Optional Profile Fields
Optional Profile Fields are enrichment fields that are not required to activate basic eligibility but improve future platform outcomes.

### 10.4 Profile Completion
Profile Completion is the state in which all mandatory fields for the relevant role have been provided and validated.

### 10.5 Profile Enrichment
Profile Enrichment refers to the completion of additional non-mandatory sections that improve context, assessment quality, searchability, and future matching.

### 10.6 Update Profile Page
The Update Profile Page is the structured screen where the user reviews and completes profile information section by section.

### 10.7 Assignment Eligibility Gate
The Assignment Eligibility Gate is the rule that prevents assignment of certain experiences, such as Assessments, Programs, or Events, until an Individual’s mandatory profile is complete.

## 11. User Roles

### 11.1 Individual
The Individual must complete mandatory personal profile fields before being eligible for Assessments, Programs, or Events.

### 11.2 Professional
The Professional must also maintain mandatory identity information and may additionally complete professional-specific fields that will later support public profile visibility and search.

### 11.3 Company
The Company profile must capture basic organization information and required location or identity data needed for platform usage.

### 11.4 SuperAdmin
The SuperAdmin may view profile completeness indirectly through downstream modules and can rely on profile readiness rules for assignment workflows.

---

## 12. Profile Strategy

### 12.1 Stepped Completion Approach
The platform shall not require every possible profile field to be completed at once.

Instead, the platform shall:
- require completion of a mandatory profile section first,
- allow all other sections to remain optional,
- and encourage users to enrich their profile progressively over time.

### 12.2 Dashboard Reminder
If a logged-in user has not completed the mandatory profile section, the dashboard shall show a visible top-level message asking the user to complete their profile.

Example intent:
- “Please complete your profile to continue.”
- “Complete your mandatory profile details before assignments can be enabled.”

### 12.3 Update Profile Redirection
Selecting the dashboard reminder shall take the user to the **Update Profile** page.

### 12.4 Enrichment Messaging
The Update Profile page shall communicate that while some fields are optional, completing them can improve platform outcomes such as better assessments.

---

## 13. Profile Types

### 13.1 Individual Profile
The Individual Profile shall include personal identity, contact, location, education, experience, and enrichment fields relevant to assessments and future recruitment support.

### 13.2 Professional Profile
The Professional Profile shall include all relevant mandatory identity fields plus professional-facing fields such as expertise, experience, professional summary, and future public-discovery-supporting fields.

### 13.3 Company Profile
The Company Profile shall include basic company information, address and location details, and other foundational organization fields needed for tenant-level identity.

## 14. Field Requirements

The following base needs have been identified:

- Name — mandatory, not editable.
- User Type — mandatory, not editable.
- Name of Company — optional.
- Address — optional.
- City — mandatory.

- Email Address — mandatory, not editable.
- Highest Degree Held — optional.

## 15. Editability Rules

The following fields shall be non-editable for user-managed updates where they are identity-controlled by the platform:

- Name
- User Type
- Phone Number
- Email Address

Other fields may be editable, subject to profile-type rules and future admin governance.

The platform should clearly distinguish:
- system-controlled identity fields,
- editable profile fields,
- and optional enrichment fields.

---

## 16. Profile Completion Rules

### 16.1 Completion Logic
A profile shall be considered complete when all mandatory fields for that role are present and valid.

### 16.2 Individual Assignment Gate
An Individual profile must be complete before the platform allows assignment of:
- Assessments,
- Programs,
- Events.

### 16.3 Optional Fields
Optional fields shall not block basic eligibility.

### 16.4 Sectioned Completion
The platform may show section-level completion status, but the key gating rule shall be based on completion of the mandatory section.

---

## 17. User Experience Requirements

### 17.1 Dashboard Prompt
When the logged-in user has not completed mandatory profile information, the dashboard shall show a prominent prompt at the top of the dashboard requesting completion.

### 17.2 Update Profile Page
The Update Profile page shall be a full structured page with multiple sections rather than a small inline edit form.

The page shall include:
- mandatory section,
- optional enrichment sections,
- profile picture upload area,
- and explanatory messaging.

### 17.3 Guidance Messaging
The Update Profile page shall show guidance explaining that optional fields are not required but can improve platform outcomes, especially assessments.

### 17.4 Save Behavior
The system shall allow users to save the profile progressively, even if optional sections remain incomplete.

### 17.5 Visual Completion Feedback
The system should show whether the mandatory section is complete and whether the user is eligible for assignments.

## 19. Business Rules

- The platform shall support a stepped profile completion approach.
- Mandatory fields must be completed before Individuals can receive assignments.
- Optional fields must not block core onboarding.
- Name, User Type, Phone, and Email shall be treated as non-editable identity fields in user-managed profile updates.
- Profile Picture shall be optional.
- Professional profiles shall support additional professional context fields.
- Company profiles shall support company-specific basic information and location fields.
- The dashboard reminder shall be shown when mandatory completion is incomplete.
- The Update Profile page shall include both mandatory and optional sections.
- The system shall explicitly encourage optional enrichment by communicating its value for better assessments.

---

## 20. Architectural Recommendation

The Profile system should be implemented as a role-aware but unified profile model.

That means:
- one canonical profile object,
- shared base fields across all roles,
- role-specific extension fields,
- and computed completion-state fields used by the dashboard and assignment workflows.

This approach is preferable to creating unrelated profile models for each role because it improves:
- consistency,
- validation reuse,
- analytics,
- extensibility,
- and downstream feature integration.

## 22. User Stories

### US-E6-01 — User Sees Dashboard Prompt for Incomplete Profile
**As a** logged-in user,  
**I want** to see a clear dashboard prompt when my mandatory profile is incomplete,  
**so that** I know I must complete it before continuing.

**Acceptance Criteria**
- Prompt appears when mandatory fields are incomplete.
- Prompt is shown prominently at the top of the dashboard.
- Prompt links to Update Profile page.

---

### US-E6-02 — User Can Open Update Profile Page
**As a** logged-in user,  
**I want** a full Update Profile page with structured sections,  
**so that** I can complete my profile in a guided way.

**Acceptance Criteria**
- Update Profile is a full page.
- Mandatory and optional sections are clearly separated.
- User can save progressive updates.

---

### US-E6-03 — Individual Must Complete Mandatory Profile Before Assignment
**As a** Product Owner,  
**I want** Individuals to complete mandatory profile fields before assignment,  
**so that** the platform has enough context before assigning experiences.

**Acceptance Criteria**
- Individuals with incomplete mandatory fields are not assignment eligible.
- Assessments, Programs, and Events cannot be assigned until profile is complete.

---

### US-E6-04 — Optional Fields Improve Profile Quality Without Blocking Usage
**As a** user,  
**I want** optional profile fields to remain non-blocking,  
**so that** I can get started without completing everything at once.

**Acceptance Criteria**
- Optional sections do not block mandatory completion.
- System messaging explains value of enrichment.

---

### US-E6-05 — Professional Can Maintain a Professional-Oriented Profile
**As a** Professional,  
**I want** extra professional fields in my profile,  
**so that** the platform can later use them for discovery and profile visibility.

**Acceptance Criteria**
- Professional profile displays additional relevant fields.
- Data is stored in role-aware structure.
- Future public profile support is not blocked.

---

### US-E6-06 — Company Can Maintain Basic Company Profile
**As a** Company,  
**I want** a basic profile with company information and location,  
**so that** the tenant identity is correctly represented in the platform.

**Acceptance Criteria**
- Company profile has company-specific fields.
- Mandatory company data can be completed and stored.

---

## 23. QA Considerations

QA should validate the following scenarios:

- dashboard reminder appears correctly for incomplete profile,
- dashboard reminder does not appear for completed mandatory profile,
- clicking reminder opens Update Profile page,
- mandatory fields are enforced by role,
- non-editable fields remain locked,
- optional fields remain editable where appropriate,
- profile picture upload works where enabled,
- progressive save works,
- completion status is calculated correctly,
- assignment eligibility changes correctly when mandatory fields become complete,
- Individuals with incomplete profile cannot be assigned Assessments, Programs, or Events,
- Professional-specific fields appear only for Professionals where applicable,
- Company-specific fields appear only for Company profiles where applicable.

---

## 24. Risks and Considerations

- If too many fields are made mandatory initially, onboarding friction will increase.
- If mandatory fields are too few, downstream modules may lack sufficient context.
- If role-specific profile fields are not modeled early, future public and recruitment use cases may require rework.
- If editability rules are unclear, identity data quality may degrade.
- If the dashboard prompt is weak or easy to ignore, profile completion rates may remain low.

---
