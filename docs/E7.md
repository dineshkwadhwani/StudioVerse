## 1. Purpose

This document defines the business and functional requirements for the **Activity Assignment** capability in StudioVerse.

The purpose of this Epic is to allow eligible logged-in users such as **Company** and **Professional** actors to assign or recommend activities to another person using a lightweight search-first workflow based on phone number or email address.

This Epic also supports a self-directed activity journey where a logged-in user can click **Register Now** or **Try Now** and assign the activity to himself or herself without entering search details.

This Epic establishes the assignment confirmation rules, wallet validation and coin deduction behavior, assignee lookup flow, self-assignment flow, assigned activity records, and post-login visibility of assigned activities for the recipient.

## 3. Executive Summary

Once logged in, a **Company** or **Professional** should be able to assign or recommend a Program, Assessment, or Event to someone else from within the product.

When the user clicks an **Assign** or **Recommend** action, the platform shall open a popup that asks for either a **phone number** or an **email address**. The user may provide either one, and the system must validate the input before performing search.

If the assignee already exists in registered users, the system shall display the user’s **phone number, email address, first name, and last name** and enable the next action button such as **Assign** or **Recommend**.

If the user is not found, the platform shall inform the assigner that the user was not found and shall request **first name** and **last name**, so that assignment can still proceed in a controlled way as a newly identified or not-yet-registered recipient.

In addition, when a logged-in user clicks **Register Now** or **Try Now** on a supported activity, the system shall follow the same assignment flow but assign the activity to the same logged-in user. In this self-assignment flow, the system shall not ask for phone number, email address, first name, or last name because the assignee is already known.

Assignment can only happen if the wallet responsible for funding the assignment has enough coins for the activity credit requirement. If insufficient coins are available, the platform must show an error message such as **“Not enough coins, please get more coins.”** If enough coins are available, the platform shall show a confirmation that **n coins will be reduced** for the assignment. For zero-credit activities, assignment may proceed without wallet blocking. Once assignment or recommendation is completed, the corresponding activity record shall be created and the recipient shall later see the activity in **My activities** after login.

## 4. Business Context

StudioVerse is designed as a structured multi-tenant platform where Programs, Assessments, and Events are not only consumed directly, but can also be assigned, recommended, or self-started by platform actors.

This means the platform needs a lightweight but reliable mechanism to:
- find a target user,
- confirm whether they are already registered,
- support controlled assignment,
- support self-assignment,
- debit the appropriate wallet,
- and make that activity visible to the assignee in their own experience.

Because the platform already uses wallet-based coin allocation for entitlement, assignment must also integrate directly with coin availability and wallet transaction tracking.

---

## 5. Problem Statement

StudioVerse currently needs a formal Activity Assignment capability that allows a Company or Professional to assign or recommend an activity to another user without navigating away into a long workflow.

The platform also needs to support a self-serve model in which a logged-in user can directly click **Register Now** or **Try Now** and receive the activity as his or her own assigned activity.

The platform also needs to handle both known and unknown assignees. Some recipients may already exist as registered users, while others may need to be identified through minimal information at the time of assignment.

Without this capability:
- activities remain isolated rather than operationally shareable,
- companies and professionals cannot direct users into the right journey,
- users cannot self-initiate supported activities consistently,
- wallet debits cannot be tied to assignment events,
- and recipients cannot see a reliable list of activities assigned to them after login.

---

## 6. Objectives

The objectives of this Epic are:

- Allow Company and Professional users to assign or recommend activities to others after login.
- Support assignment and recommendation flows from Programs, Assessments, and Events.
- Support self-assignment through **Register Now** and **Try Now** for supported activity types.
- Open a popup-based flow when the Assign / Recommend action is clicked.
- Allow assignee search by **phone number** or **email address** for assignment to others.
- Validate search input before lookup.
- Display first and last name when a registered user is found.
- Allow not-found users to be identified through first name, last name, and whichever contact field (phone or email) was not part of the original search.
- Create an **Assigned Activity** record upon successful assignment.
- Enforce wallet balance validation before assignment.
- Show insufficient wallet messaging when coins are not enough.
- Show confirmation before deducting coins.
- Record the wallet transaction linked to the assignment.
- Show assigned activities to the assignee after login.
- Provide a wallet transaction view link from **My Wallet**.

## 10. Domain Definitions

### 10.1 Activity
An Activity is a platform object that may be assigned, recommended, or self-registered by a user, such as an Assessment, Program, or Event.

### 10.2 Assigned Activity
An Assigned Activity is the record created when one user directs an activity to another user or to himself / herself through an assignment, recommendation, or self-registration action.

### 10.3 Assigner
The Assigner is the logged-in Company or Professional user who initiates the assignment or recommendation.

### 10.4 Assignee
The Assignee is the target user to whom the activity is assigned or recommended.

### 10.5 Self-Assignment
Self-Assignment is the flow in which a logged-in user clicks **Register Now** or **Try Now** and the activity is assigned to the same user.

### 10.6 Recommendation
A Recommendation is a lighter form of directed activity suggestion. In UI terms it may share much of the same flow as assignment, though the downstream status may differ.

### 10.7 Activity Credit Requirement
The Activity Credit Requirement is the number of coins required to complete the assignment of a given activity.

### 10.8 Wallet Transaction
A Wallet Transaction is the ledger record created when coins are debited due to a successful assignment.

## 11. User Roles

### 11.1 Company
A Company user may assign or recommend supported activities to another user after login.

### 11.2 Professional
A Professional user may assign or recommend supported activities to another user after login.

### 11.3 Logged-in Self User
A logged-in eligible user may click **Register Now** or **Try Now** and assign the activity to himself or herself.

### 11.4 Assignee / Individual
The Assignee is the person who receives the assigned activity and later sees it after login.

### 11.5 SuperAdmin
The SuperAdmin may rely on this assignment model indirectly through wallet, activity, and reporting layers, but the core assignment action described here is primarily performed by Company and Professional users and self-directed users.

## 13. Assignment Entry Points

### 13.1 Action Buttons
Supported activity screens shall expose action buttons such as:
- **Assign**
- **Recommend**
- **Register Now**
- **Try Now**

The exact label may depend on activity type or business context.

### 13.2 Popup Launch for Assign / Recommend
When the user clicks **Assign** or **Recommend**, the platform shall open a popup / modal for assignee lookup and assignment processing.

### 13.3 Direct Self-Assignment Flow
When the user clicks **Register Now** or **Try Now**, the platform shall initiate the same assignment workflow but with the logged-in user pre-selected as the assignee.

The system may either:
- open a simplified confirmation modal, or
- directly move to wallet confirmation if no additional input is required.

The system shall not ask for phone number, email address, first name, or last name in self-assignment mode.

---

## 14. Popup Workflow for Assignment to Others

### 14.1 Search Inputs
The popup shall include fields for:
- phone number,
- email address.

The assigner must provide at least one of these values.

### 14.2 Validation Rules
The system shall validate:
- phone number format if phone is provided,
- email format if email is provided,
- and ensure at least one of the two fields is present before search is executed.

### 14.3 Search Action
When the user clicks **Search**, the system shall look for an existing registered user matching the provided phone number or email address.

### 14.4 Found State
If a user is found, the popup shall:
- display first name,
- display last name,
- identify that the target user is recognized,
- and enable the next action button such as **Assign** or **Recommend**.

### 14.5 Not Found State
If a user is not found, the popup shall show a message such as:
- **“User not found. Please provide first name, last name, and the missing contact detail.”**

The popup shall then require:
- first name,
- last name,
- and the missing contact field (email if phone was searched, or phone if email was searched).

This allows the flow to proceed with a minimally identified assignee record.

---

## 15. Self-Assignment Flow

### 15.1 Trigger
When a logged-in user clicks **Register Now** or **Try Now**, the system shall initiate self-assignment for that activity.

### 15.2 Assignee Resolution
In self-assignment mode, the assignee shall automatically be the currently logged-in user.

### 15.3 No Search Input Required
The system shall not ask for:
- phone number,
- email address,
- first name,
- last name.

### 15.4 Shared Downstream Logic
After assignee resolution, self-assignment shall follow the same downstream flow as standard assignment, including:
- activity eligibility logic,
- wallet sufficiency check,
- coin deduction confirmation,
- assigned activity creation,
- wallet transaction creation,
- and visibility in assigned activities.

### 15.5 Activity Scope
For first release, self-assignment is required for:
- Assessments via **Try Now** or **Register Now** as applicable,
- Programs via **Register Now**,
- Events via **Register Now**.

---

## 16. Wallet and Coin Validation

### 16.1 Credit Requirement
Each assignable activity shall define or expose the number of coins required for assignment.

### 16.2 Wallet Sufficiency Check
Before assignment is finalized, the system shall check whether the wallet responsible for funding that action has enough available coins for that activity.

### 16.3 Insufficient Coins
If there are not enough coins, the system shall show a message such as:
- **“Not enough coins, please get more coins.”**

In this case:
- assignment shall not proceed,
- wallet balance shall not change,
- no wallet transaction shall be recorded,
- and no assigned activity record shall be created.

### 16.4 Sufficient Coins Confirmation
If enough coins are available, the system shall show a confirmation stating that **n coins will be reduced from the wallet** for the assignment. If the activity requires zero coins, the system may proceed without a wallet-debit warning.

### 16.5 Debit and Recording
Upon confirmation and successful assignment:
- the coins shall be debited,
- a wallet transaction shall be created,
- and the assigned activity record shall be saved.

---

## 17. Wallet Ownership Clarification

For this Epic, the wallet check and deduction must be applied against the wallet that business rules define as responsible for funding the assignment.

The default recommendation for first implementation is:
- deduct coins from the **assigner-owned wallet** when that assigner is directly funding the activity assignment,
- and deduct coins from the **logged-in user wallet** during self-assignment through **Register Now** or **Try Now**.

If business rules instead require deduction from the **assignee wallet** in assign-to-other scenarios, that rule must be modeled explicitly and consistently across wallet and assignment services.

Implementation must not leave wallet ownership ambiguous.

## 20. User Experience Requirements

### 20.1 Modal Design
The assignment popup shall:
- be lightweight,
- guide the user clearly through search,
- expose state transitions cleanly,
- and avoid forcing navigation away from the source activity page.

### 20.2 Self-Assignment UX
The self-assignment flow through **Register Now** or **Try Now** shall be faster and simpler than assignment to another user because the assignee is already known.

### 20.3 Conditional UI States
The popup or flow shall show different UI states for:
- initial entry,
- invalid input,
- searching,
- user found,
- user not found,
- self-assignment ready state,
- insufficient coins,
- confirmation,
- success.

### 20.4 Clear Messaging
The system shall communicate clearly:
- whether the user was found,
- whether additional name data is needed,
- whether the activity is being assigned to self,
- whether wallet balance is insufficient,
- how many coins will be reduced,
- and whether assignment succeeded.

## 21. Data Requirements

### 21.1 Assignment Search Input Model
The popup shall support:
- `phone`
- `email`
- `firstName` (required only if user not found)
- `lastName` (required only if user not found)

### 21.2 Assignment Transaction Linkage
Assignment records should support references to:
- wallet transaction,
- source activity,
- assigner,
- assignee,
- tenant,
- and assignment outcome.

### 21.3 Wallet Transaction Metadata
The wallet transaction created for assignment should include:
- transaction type,
- debit amount,
- source activity type,
- source activity id,
- assignment id,
- actor responsible,
- and timestamp.

## 22. Business Rules

- Company and Professional users may assign or recommend supported activities after login.
- Logged-in users may self-assign supported activities through **Register Now** or **Try Now**.
- The assigner must provide at least one of phone number or email address before search when assigning to another user.
- Search input must be validated before user lookup.
- If a registered user is found, the system shall display first and last name and enable assignment action.
- If a registered user is not found, the system shall request first name and last name before proceeding.
- In self-assignment mode, the system shall not request phone number, email, first name, or last name.
- Assignment, recommendation, or self-registration shall create an Assigned Activity record.
- Assignment shall proceed only if the relevant wallet has sufficient coins.
- If wallet coins are insufficient, assignment shall not happen.
- If wallet coins are sufficient, the system shall show a coin deduction confirmation before finalizing.
- Successful assignment shall create a wallet transaction.
- The recipient shall see assigned activities after login.
- My Wallet shall provide access to wallet transaction history.

---

## 23. Architectural Recommendation

The Activity Assignment system should be implemented as a reusable cross-module service rather than embedding assignment logic separately inside Program, Assessment, and Event pages.

The recommended architecture is:
- one common assignment modal,
- one shared assignee lookup service,
- one shared self-assignment flow,
- one shared wallet validation and debit flow,
- one canonical Assigned Activity entity,
- and activity-type-specific adapters for Programs, Assessments, and Events.

This approach improves:
- consistency,
- maintainability,
- auditability,
- reuse,
- and future extension to more activity types.

## 27. Risks and Considerations

- If wallet ownership rules are ambiguous, incorrect wallet debits may occur.
- If user lookup matching is weak, assignment could target the wrong user.
- If not-found assignees are not modeled carefully, duplicate user identities may increase later.
- If self-assignment is implemented separately from common assignment logic, behavior may drift across modules.
- If assignment logic is duplicated separately across activities, maintenance cost will rise.
- If wallet debit and assignment creation are not transaction-safe, data inconsistency can occur.
