## 1. Purpose

This document defines the business and functional requirements for the **Access Control** capability in StudioVerse.

The purpose of this Epic is to ensure that each logged-in user sees only the menus, pages, and high-level actions that are permitted for that user’s role, based on the role matrix defined for **Super Admin, Company, Professional, and Individual** users.

This Epic also establishes the role-based navigation model, page-level authorization rules, menu rendering behavior, denial handling, and extensibility for future permission refinement.

## 3. Executive Summary

StudioVerse supports multiple user roles with clearly different responsibilities, so the platform must not show the same menus and capabilities to every logged-in user.

Based on the revised access matrix, **Super Admin** has the broadest administrative access, **Company** has organization-scoped management access, **Professional** has a more limited coach-scoped operational access, and **Individual** has a restricted self-service access model.

This Epic defines which menus are visible to which roles, what each menu means operationally, and how the system must enforce access both in navigation and in backend authorization checks.

---

## 4. Business Context

StudioVerse is a multi-role platform spanning tenant administration, professional services, individual participation, programs, assessments, events, wallets, assigned activities, and activity visibility.

Because these roles are intentionally different, the platform needs explicit role-based access control so users cannot access features outside their responsibility area, such as tenant management, platform-wide user management, or cross-role creation workflows.

This is especially important for menu-driven applications, where incorrect visibility can create confusion, security issues, and accidental access to privileged operations.

---

## 5. Problem Statement

StudioVerse currently needs a formal Access Control capability that determines which menus and pages each user role can access after login.

The platform must align the visible navigation to the approved role matrix and also prevent unauthorized access if a user attempts to open a page directly by URL or indirect navigation.

Without this capability:
- users may see irrelevant or unauthorized menus,
- role boundaries may become inconsistent across modules,
- privileged actions may be exposed incorrectly,
- and product behavior may differ across frontend and backend implementations.

---

## 6. Objectives

The objectives of this Epic are:

- Define a canonical access model for Super Admin, Company, Professional, and Individual users.
- Ensure menu visibility is role-based and aligned with the revised access table.
- Enforce page-level authorization in addition to menu visibility.
- Restrict module actions according to role scope.
- Prevent unauthorized access through direct URLs or backend requests.
- Keep the authorization model extensible for future fine-grained permissions.
- Ensure access behavior is consistent across web UI, APIs, and future modules.
- Support stakeholder clarity by documenting each menu’s intended role scope.

---

## 10. Domain Definitions

### 10.1 Access Control
Access Control is the set of rules that determines which menus, pages, and actions a logged-in user may see or use.

### 10.2 Role
A Role is the primary access identity of a user in StudioVerse, such as **Super Admin**, **Company**, **Professional**, or **Individual**.

### 10.3 Menu Authorization
Menu Authorization is the rule that determines whether a menu item is visible in the application navigation for a given role.

### 10.4 Page Authorization
Page Authorization is the rule that determines whether a user may open a given page or route even if they try to access it directly.

### 10.5 Action Authorization
Action Authorization is the rule that determines whether a user can perform a specific action inside an accessible module, such as creating, assigning, promoting, managing, or viewing records.

### 10.6 Scope Restriction
Scope Restriction defines the boundaries within which a role can manage data, such as:
- all platform users,
- only users in a company,
- only individuals assigned to a professional,
- or only the logged-in user’s own records.

### 10.7 My Activities
**My activities** is the user-facing area where a logged-in user can see activities assigned or relevant to himself / herself based on role and platform behavior.

---

## 11. User Roles

### 11.1 Super Admin
The Super Admin has the broadest platform access and can manage platform-wide entities including all users, tenants, assessments, coins, programs, events, and activity assignment.

### 11.2 Company
A Company user has organization-level access limited to company-relevant modules such as company-scoped users, wallet management, programs, events, cohorts, individuals, assigned activity operations, and activity visibility.

### 11.3 Professional
A Professional user has limited operational access focused on managed individuals, self-wallet, self-created programs and events, self cohorting, activity assignment, and activity visibility.

### 11.4 Individual
An Individual user has a self-service access model focused on dashboard, profile, wallet, self-assignment capability, and personal activity visibility.

---

## 12. Access Strategy

### 12.1 Role-Based Visibility
The system shall render menus based on the logged-in user’s role and only show menu items permitted for that role.

### 12.2 Defense in Depth
The system shall enforce authorization at both:
- frontend navigation level,
- backend page / API level.

Hiding a menu alone shall not be treated as sufficient security.

### 12.3 Consistent Role Semantics
The meaning of each role must remain consistent across all modules so that, for example, a Professional cannot receive broader user-management rights in one module than in another.

## 15. Action Restrictions

### 15.1 Manage Users
- Super Admin can manage all platform users.
- Company can manage only Professionals and Individuals within the company.
- Professional can manage only Individuals assigned to that coach.
- Individual cannot access Manage Users.

### 15.2 Manage Coins / Manage Wallet
- Super Admin can manage platform coin operations broadly.
- Company can manage, buy, or allocate coins for the company.
- Professional can only buy and use coins for self.
- Individual can only buy and use coins for self.

### 15.3 Manage Programs
- Super Admin can manage programs fully.
- Company can create own programs but cannot mark a program as **Promote**.
- Professional can create own programs but cannot mark a program as **Promote**.
- Individual cannot access Manage Programs.

### 15.4 Manage Events
- Super Admin can manage events fully.
- Company can create own events but cannot mark an event as **Promote**.
- Professional can create own events but cannot mark an event as **Promote**.
- Individual cannot access Manage Events.

### 15.5 Manage Cohort
- Company can create a cohort and assign a coach to that cohort.
- Professional can create a cohort for self.
- Super Admin and Individual cannot access Manage Cohort.

### 15.6 Manage Individual
- Company can add individuals to the company.
- Professional can add individuals to self.
- Super Admin and Individual cannot access Manage Individual.

### 15.7 Manage Assessment
- Only Super Admin can access Manage Assessment.
- Company, Professional, and Individual cannot access Manage Assessment.

### 15.8 Assign Activity
- Super Admin can assign activity.
- Company can assign activity.
- Professional can assign activity.
- Individual can access Assign Activity but may assign only to self.

### 15.9 My Activities
- Company can access My activities.
- Professional can access My activities.
- Individual can access My activities.
- Super Admin cannot access My activities in this version.

---

## 16. Page Authorization Rules

### 16.1 Menu Visibility Is Not Enough
Even if a menu is hidden, the platform shall still block unauthorized direct access to the corresponding route or API.

### 16.2 Unauthorized Access Response
If a user tries to access a page that is not permitted for the role, the system shall:
- deny access,
- avoid exposing restricted data,
- and route the user to an appropriate fallback page such as Dashboard or an Access Denied page.

### 16.3 Backend Enforcement
APIs and server-side actions shall validate role and scope before returning data or processing updates.

---

## 17. Navigation Behavior

### 17.1 Dynamic Menu Rendering
Menus shall be rendered dynamically based on the authenticated user role.

### 17.2 Role-Aware Labels
Menu labels should remain consistent across roles even when permissions differ, so users see a stable navigation language.

### 17.3 Empty-State Avoidance
A user shall not be shown a menu item that leads to an empty page only because they lack access to underlying actions.

---

## 18. Data and Configuration Requirements

### 18.1 Role Source
The user’s active role shall be available from the authenticated user profile / identity model.

### 18.2 Permission Model
The system should support a configurable permission map such as:

- role
- menu key
- route key
- visible
- allowed actions
- scope type

### 18.3 Scope Types
The permission structure should support scope values such as:
- `platform_all`
- `tenant_only`
- `assigned_individuals_only`
- `self_only`

These values align with the distinctions shown in the role matrix.

---

## 19. Business Rules

- Every logged-in user shall see Dashboard and Update Profile.
- Only Super Admin shall see Manage Tenants.
- Only Super Admin shall see Manage Assessment.
- Company and Professional shall see Manage Users with restricted scope.
- Individual shall not see Manage Users.
- Company, Professional, and Individual shall see Manage Coins / Manage Wallet, but scope differs by role.
- Company and Professional may create Programs and Events but may not mark them as Promote.
- Individual shall not see Manage Programs or Manage Events.
- Only Company and Professional shall access Manage Cohort and Manage Individual.
- All roles may access Assign Activity, but Individual is restricted to self-assignment only.
- My activities shall be visible only to Company, Professional, and Individual.

---

## 20. Architectural Recommendation

The Access Control system should be implemented as a shared authorization layer rather than hardcoding menu conditions separately in each page.

The recommended approach is:
- one canonical role-permission configuration,
- one reusable menu rendering service,
- one route guard layer,
- one backend authorization middleware,
- and one scope evaluation model for entity-level restrictions.

This approach improves:
- consistency,
- maintainability,
- security,
- traceability,
- and future extensibility.

## 22. User Stories

### US-E8-01 — Role-Based Menus Are Visible Correctly
**As a** logged-in user,  
**I want** to see only the menus relevant to my role,  
**so that** the application feels appropriate to my responsibility.

**Acceptance Criteria**
- Menus match the approved role matrix.
- Unauthorized menus are hidden.

---

### US-E8-02 — Unauthorized Pages Are Blocked
**As a** Product Owner,  
**I want** page access to be blocked even when a user tries direct navigation,  
**so that** restricted modules remain secure.

**Acceptance Criteria**
- Unauthorized routes are blocked.
- User is redirected or shown Access Denied.

---

### US-E8-03 — Company Access Is Tenant Scoped
**As a** Company user,  
**I want** my management access limited to my company scope,  
**so that** I cannot operate outside my organization.

**Acceptance Criteria**
- Company can manage only Professionals and Individuals in its company.
- Company cannot access tenant-wide administration.

---

### US-E8-04 — Professional Access Is Coach Scoped
**As a** Professional user,  
**I want** my user-management access limited to my assigned individuals,  
**so that** I only operate within my coaching scope.

**Acceptance Criteria**
- Professional manages only assigned individuals.
- Professional cannot access platform-wide or company-wide user data.

---

### US-E8-05 — Individual Access Is Self-Service Only
**As an** Individual user,  
**I want** only self-service menus and self-assignment capability,  
**so that** I cannot access administrative modules.

**Acceptance Criteria**
- Individual sees only permitted menus.
- Individual can assign activity only to self.

---

### US-E8-06 — My Activities Is Visible Only to Applicable Roles
**As a** Product Owner,  
**I want** My activities to be visible only for the roles that need activity visibility,  
**so that** navigation remains relevant and role-correct.

**Acceptance Criteria**
- Company can access My activities.
- Professional can access My activities.
- Individual can access My activities.
- Super Admin cannot access My activities.

## 23. QA Considerations

QA should validate the following scenarios:

- each role sees the correct menus after login,
- hidden menus are not rendered,
- direct URL access to hidden modules is blocked,
- API requests to unauthorized modules are rejected,
- Company user management is limited to company scope,
- Professional user management is limited to assigned individuals,
- Individual cannot access admin modules,
- Promote action is hidden or blocked for Company and Professional in Programs and Events,
- Manage Cohort appears only for Company and Professional,
- Manage Assessment appears only for Super Admin,
- My activities appears only for Company, Professional, and Individual,
- and Assign Activity is restricted to self-only for Individual users.

---

## 24. Risks and Considerations

- If menu visibility and backend authorization are implemented separately without a common source of truth, permission drift may occur.
- If scope restrictions are not enforced carefully, users may see or update data outside their intended boundary.
- If direct URL blocking is missed, hidden menus will create a false sense of security.
- If Promote restrictions are not enforced server-side, Company or Professional users may incorrectly elevate content.
- If My activities visibility is not aligned with assignment logic, users may see inconsistent activity states.

---

## 25. Definition of Done

This Epic is considered complete when:

- role-based menu visibility matches the approved matrix,
- unauthorized pages are blocked,
- backend authorization validates role and scope,
- Company and Professional scope restrictions are enforced,
- Individual access is limited to self-service capabilities,
- Manage Assessment is restricted to Super Admin,
- My activities is visible only to Company, Professional, and Individual,
- Promote restrictions work correctly for Programs and Events,
- and Product, Engineering, and QA approve the access behavior.
