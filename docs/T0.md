# T0 — Project Setup & Development Environment

Status: Working markdown version for Copilot context.  
Canonical source: T0 technical foundation spec and Technical Stories Master.

## Objective

Establish the initial engineering baseline for the StudioVerse codebase so development can proceed in a clean, consistent, and reproducible way.

T0 should ensure that developers can clone the repo, install dependencies, configure the environment, run the project locally, follow agreed standards, and contribute through a shared workflow.[file:252][file:181]

## Scope

Epic T0 covers:
- repository creation
- branching strategy
- local development setup guide
- dependency installation
- Next.js project initialization
- TypeScript setup
- ESLint and Prettier setup
- environment variable conventions
- README initialization
- commit message conventions

This epic is foundational. It does not deliver user-facing product functionality.[file:252][file:181]

## Purpose

T0 exists to reduce early engineering chaos.

Without this baseline, later work tends to suffer from inconsistent local setups, unclear collaboration rules, weak code quality discipline, onboarding friction, and avoidable rework. T0 creates the delivery foundation that later technical and functional epics depend on.[file:252][file:181]

## In scope

T0 should establish:
- one shared repository for the platform
- one agreed branch workflow
- one reproducible local setup path
- one standardized dependency baseline
- one initialized Next.js application
- one strict TypeScript setup
- one linting and formatting baseline
- one documented environment-variable model
- one developer-facing README
- one consistent commit convention

## Out of scope

Do not use T0 to implement:
- Firebase project provisioning
- Firestore collections
- authentication flows
- dashboards
- deployment pipelines
- security rules
- RBAC logic
- business features
- product page implementation

Those belong to later technical or functional epics.[file:252]

## Repository rules

Create a single repository for the StudioVerse platform.

Working expectations:
- repository name: `studioverse`
- visibility: private
- initial commit exists
- authorized development team members have access

This repository is intended to house the shared codebase for all three Studio deployments.[file:181]

## Branching strategy

Use a documented branching model that supports team collaboration and controlled releases.

Branch model:
- `main` — production-ready only
- `staging` — staging-ready integration branch
- `dev` — active development branch
- `feature/[story-id]-[short-description]`
- `fix/[story-id]-[short-description]`
- `hotfix/[description]`

Rules:
- no direct commits to `main` or `staging`
- merges to `main` require PR review
- branch names should follow convention consistently
- stale merged branches should be cleaned up regularly

This branching strategy is already defined in the technical stories reference and should be treated as the default team workflow.[file:181]

## Next.js initialization

Initialize the application with Next.js App Router and a scalable baseline.

Project setup should include:
- App Router enabled
- TypeScript enabled
- `src/` directory structure
- Tailwind configured at init
- `@/*` path alias configured
- no static-export-only setup

`next.config.js` should remain compatible with SSR/server components and allow future routing/domain requirements.[file:181]

## TypeScript rules

TypeScript is the standard language layer for the codebase.

Required baseline:
- `strict: true`
- `noImplicitAny: true`
- `strictNullChecks: true`
- `moduleResolution: bundler`
- `@/*` path aliases
- target `ES2017` or later

Rules:
- avoid `any` unless justified
- avoid `@ts-ignore` unless justified
- keep shared types organized in `src/types/` or local module type files
- make type-checking part of the normal workflow

This ensures later epics are built on a strongly typed codebase.[file:181]

## ESLint and Prettier

Set code quality and formatting standards early.

ESLint baseline:
- use `eslint-config-next`
- include unused variable and production logging discipline
- support TypeScript-aware linting

Prettier baseline:
- print width 100
- single quotes
- no semicolons
- tab width 2
- trailing commas where configured
- no config conflicts with ESLint

Recommended extras:
- format-on-save in VS Code settings
- pre-commit hooks using husky + lint-staged

The goal is to make clean code the default instead of a cleanup step later.[file:181]

## Environment variables

Define and document a safe environment-variable strategy from the beginning.

Categories:
- Studio identity for app shell
- Firebase client config
- general app config
- backend/server-only secrets

Important public variables include:
- `NEXT_PUBLIC_STUDIO_TYPE`
- `NEXT_PUBLIC_APP_ENV`
- `NEXT_PUBLIC_APP_BASE_URL`
- Firebase public config values
- `NEXT_PUBLIC_ENABLE_MOCK_SESSION`

Server-only values include:
- `FIREBASE_SERVICE_ACCOUNT_KEY`
- `GROQ_API_KEY`
- `RESEND_API_KEY`
- `STRIPE_SECRET_KEY` later

Rules:
- only safe values use `NEXT_PUBLIC_`
- secrets must never be exposed in browser-shipped config
- `.env.local` must be gitignored
- `.env.example` must be committed with keys only, no secrets

The technical stories also clarify that `NEXT_PUBLIC_STUDIO_TYPE` is for the authenticated app shell and not for public landing-page switching.[file:181]

## Commit convention

Use Conventional Commits.

Format:
- `type(scope): description`

Common types:
- `feat`
- `fix`
- `chore`
- `docs`
- `refactor`
- `test`
- `style`
- `ci`

Examples:
- `feat(auth): implement Google SSO login`
- `fix(assignments): correct overdue detection timezone handling`
- `chore(deps): update firebase version`

This keeps history readable and supports cleaner release and CI/CD workflows later.[file:181]

## Risks to avoid

Common risks T0 is meant to prevent:
- inconsistent local developer setups
- unclear branching and merge practices
- missing setup documentation
- weak linting/formatting standards
- hardcoded local config and secrets
- avoidable onboarding delays

Mitigation approach:
- keep conventions simple
- document everything critical
- validate setup on more than one machine
- enforce standards before feature work accelerates

These concerns are explicitly highlighted in the formal T0 foundation spec.[file:252]

## Non-functional expectations

The setup created in T0 should emphasize:
- maintainability
- consistency
- readability
- reproducibility

- scalability for later epics

The baseline should be professional and lightweight, not over-engineered, but strong enough to support long-term product development.[file:252]

## Delivery guidance

Treat T0 as the first quality gate for the entire codebase.

If T0 is clean, later epics move faster and with fewer avoidable issues. If T0 is weak, every later epic inherits unnecessary friction.[file:252][file:181]
