# T1 — Firebase & Backend Setup

Status: Working markdown version for Copilot context.  
Canonical source: T1 technical foundation spec and Technical Stories Master.

## Objective

Establish the core Firebase backend foundation for StudioVerse.

T1 should ensure Firebase projects exist for each environment, Firestore/Auth/Storage/Functions are enabled, baseline backend security rules are in place, local emulation works, Firebase CLI targets are configured, core collections are defined, and backend logging foundations are established.[file:251][file:181]

## Scope

Epic T1 covers:
- Firebase project setup
- Firestore enablement
- baseline security rules
- Firebase Authentication setup
- Firebase Storage setup
- Firebase Functions setup
- Firebase CLI configuration
- Firebase Emulator Suite setup
- baseline collection/index definitions
- backend logging foundation

This epic creates backend infrastructure, not end-user business workflows.[file:251]

## Purpose

T1 is the backend platform-enablement layer for StudioVerse.

Because later epics depend on identity, persistence, trusted server-side execution, file storage, and secure local development, the backend foundation must exist before feature work scales. T1 reduces the risk of fragmented backend decisions later.[file:251]

## Strategic principles

The backend foundation should be:
- security-conscious
- modular
- emulator-friendly
- environment-aware
- aligned with long-term data model growth
- easy for developers to run locally
- structured for trusted backend logic

These goals are explicit in the T1 foundation specification.[file:251]

## Environment model

StudioVerse should use three Firebase projects, one per environment:
- `studioverse-dev`
- `studioverse-staging`
- `studioverse-prod`

This environment split is already reflected in the technical stories and should remain the default operating model for backend isolation.[file:181]

## Project setup

Per environment, Firebase should have:
- Firestore
- Authentication
- Storage
- Functions
- optional Hosting, though Vercel remains the main frontend host

Region should be consistent across all projects, with the technical stories recommending `asia-south1` for an India/UAE user base or `us-central1` if global latency is preferred.[file:181]

## Firestore

Firestore is the primary structured database for the platform.

Requirements:
- Native mode, not Datastore mode
- region aligned across all environments
- initial locked posture before broader rule expansion
- ready for later multi-tenant collection structure

The technical stories specifically say the initial rules posture should be deny-all until the baseline rules story is completed.[file:251][file:181]

## Security rules baseline

The initial backend rules foundation should enforce:
- authentication required for reads and writes
- no unauthenticated collection access
- `studioType` scoping on tenant collections
- placeholder coverage for all 25 collections until T4 expands them

Minimum expected outcomes:
- `firestore.rules` is committed
- rules deploy to dev successfully
- unauthenticated access fails
- authenticated access to a user’s own document is allowed
- emulator rule tests exist for critical paths[ file:181]

## Authentication

Firebase Authentication should be enabled as the identity layer.

Required providers:
- Email/Password
- Google Sign-In

Important auth setup notes:
- email enumeration protection enabled
- authorized domains configured for dev, staging, and prod
- Studio type is stored in the Firestore `users` document, not in Firebase Auth custom claims
- all three Studio types share Auth within the same project per environment[ file:181][file:251]

## Storage

Firebase Storage should be enabled for future files and media.

Primary use cases already identified:
- user avatars
- program module assets
- AI-generated report PDFs
- company and Studio logos

Baseline storage behavior:
- users can access their own designated files
- reports are written by Functions, not directly by the UI
- storage rules start simple and get tightened further in T4[ file:181]

## Storage paths

Expected path patterns:
- `/avatars/{userId}/profile.jpg`
- `/programs/{programId}/modules/{moduleId}/{filename}`
- `/reports/{userId}/{toolId}/{reportId}.pdf`
- `/companies/{companyId}/logo.jpg`

These paths are useful for keeping future uploads consistent and predictable across frontend and backend work.[file:181]

## Functions

Firebase Functions are the trusted business logic layer.

Setup expectations:
- Cloud Functions Gen 2
- Node.js 20 runtime
- TypeScript
- same region as Firestore
- root `functions/` directory
- modular folder structure by domain

The functions layer exists to centralize trusted operations instead of pushing sensitive logic into the client.[file:181][file:251]

## Initial Function stubs

The technical stories already define these initial deployable stubs:
- `buildUserContext`
- `onToolSubmitted`
- `generateReport`
- `sendNotificationEmail`
- `detectOverdueAssignments`
- `onCohortMemberAdded`

These stubs matter because they signal the intended backend responsibility model early, even before full business logic exists.[file:181]

## Functions structure

Expected structure:

```text
functions/
  src/
    index.ts
    auth/
      buildUserContext.ts
    assignments/
      detectOverdue.ts
      onCohortMemberAdded.ts
    reports/
      onToolSubmitted.ts
      generateReport.ts
    notifications/
      sendEmail.ts
    types/
      index.ts
    utils/
      firestore.ts
      validation.ts
  package.json
  tsconfig.json
  .eslintrc.js
```

This modular shape should stay aligned with the broader service/module architecture rather than becoming a dumping ground for backend code.[file:181]

## Firebase CLI

Firebase CLI should be fully configured for all environments.

Expected `.firebaserc` project aliases:

```json
{
  "projects": {
    "dev": "studioverse-dev",
    "staging": "studioverse-staging",
    "default": "studioverse-prod"
  }
}
```

This gives the team a predictable environment-switching model for local admin and deploy workflows.[file:181]

## Firebase config files

`firebase.json` should define:
- Firestore rules targets
- Firestore indexes targets
- Functions deploy target
- Storage rules target
- Hosting targets if needed
- emulator configuration

CLI behavior should support:
- `firebase use dev`
- `firebase use staging`
- `firebase use default`
- targeted deploys like `firebase deploy --only functions`[ file:181]

## Emulator Suite

Local development should not depend on live Firebase environments.

Emulators expected:
- Auth on 9099
- Firestore on 8080
- Functions on 5001
- Storage on 9199
- Emulator UI on 4000

This is a critical part of T1 because it enables safe local backend testing without touching shared cloud data.[file:181][file:251]

## Emulator behavior

Application-side emulator detection should work like this:
- when `NEXT_PUBLIC_APP_ENV=dev` and the app is running locally, Firebase SDK connects to emulators
- emulator connection logic lives in `src/lib/firebase.ts`

Recommended outcome:
- `firebase emulators:start` runs without port conflicts
- local app flows work against emulated services
- optional seed data can speed up testing[ file:181]

## Collections baseline

T1 expects the platform’s baseline Firestore collection structure to be identified in a forward-compatible way.

The technical stories list 25 collections:
- `users`
- `userContexts`
- `companies`
- `companyMembers`
- `professionalProfiles`
- `professionalRelations`
- `cohorts`
- `cohortMembers`
- `programs`
- `programModules`
- `programProgress`
- `tools`
- `toolQuestions`
- `toolSubmissions`
- `toolResults`
- `assignments`
- `events`
- `eventRegistrations`
- `reports`
- `notifications`
- `widgetSessions`
- `auditLogs`
- `studioConfig`
- `marketplace`
- `payments`[file:181]

## Collection purpose

These collections together support identity, RBAC context, tenant relationships, learning content, assessments, assignments, events, reports, audit history, Studio runtime config, marketplace flows, and payment records. Their names and purposes should stay stable because later rules, indexes, and services will depend on them.[file:181][file:251]

## Index baseline

The technical stories already define required composite indexes including:
- `assignments`: `studioType`, `professionalId`, `status`, `dueDate`
- `assignments`: `studioType`, `individualId`, `status`
- `programs`: `studioType`, `status`, `tags ARRAY_CONTAINS`
- `tools`: `studioType`, `status`
- `cohortMembers`: `cohortId`, `status`
- `notifications`: `userId`, `isRead`, `createdAt DESC`
- `auditLogs`: `studioType`, `actorId`, `createdAt DESC`

These should live in `firestore.indexes.json` and deploy cleanly to dev first.[file:181]

## Logging foundation

T1 includes backend logging setup as a baseline operational requirement.

Expected logging posture:
- frontend logger utility in `src/lib/logger.ts`
- browser console logging in dev
- reduced/suppressed production console noise
- error monitoring provider for staging/prod
- Firebase Functions should use `functions.logger`, not raw `console.log`

Logging is treated as part of setup, not a later optional improvement.[file:181][file:251]

## Functions logging format

Functions-side logging should be structured and traceable.

The technical stories recommend a log shape containing:
- `level`
- `message`
- `studioType`
- `userId`
- `functionName`
- `timestamp`

Errors should be caught, logged, and then rethrown in a controlled way.[file:181]

## Risks to prevent

T1 is intended to reduce:
- insecure default backend setup
- poor collection design that causes rework
- lack of emulator usage
- unstructured Functions code
- weak operational traceability

Mitigation strategy:
- apply rules early
- keep collections aligned to the master data model
- use emulators from the start
- keep Functions modular
- establish logging as part of setup[ file:251]
