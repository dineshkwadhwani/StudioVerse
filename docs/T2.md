# T2 — Frontend Architecture

Status: Working markdown version for Copilot context.  
Canonical source: Technical Stories Master and architecture docs.

## Objective

Define the frontend architecture foundations for the StudioVerse codebase.

T2 establishes the folder structure, routing conventions, module patterns, shared component rules, state management approach, service layer boundaries, constants, hooks, and content-configuration model that all later functional epics will depend on.[file:181]

## Scope

Epic T2 covers:
- frontend folder structure
- App Router route conventions
- root and app-shell layout foundation
- module/component patterns
- Zustand store setup
- service layer setup
- constants and enums
- shared hooks/utilities conventions
- form architecture baseline
- landing-page content config structure

This epic is not about shipping end-user features. It is about setting the engineering patterns that later epics must follow.[file:181]

## Purpose

T2 is the architectural contract for frontend development.

Its job is to ensure that all later work lands in a consistent structure, uses the same routing and data-access rules, avoids direct backend leakage into UI components, and remains maintainable as the three-Studio platform grows.[file:181]

## In scope

T2 should define and create:
- canonical `src/` folder structure
- public landing page directories for each Studio
- shared authenticated `(app)` shell route group
- feature module patterns under `src/modules/`
- shared config/constants/hooks/services/stores/types layout
- route constants
- state management stores
- service stubs
- form and component conventions
- content-config structure for studio landing pages

## Out of scope

Do not use T2 to implement:
- final business features
- full dashboard functionality
- final auth flows
- real admin workflows
- direct feature-specific Firestore logic in UI
- ad hoc route patterns that bypass the architecture
- one-off components that ignore the module pattern

T2 should create the structure that later epics build on, not bypass it.[file:181]

## Core architecture rules

Follow these rules throughout frontend implementation:
- use Next.js App Router
- keep all code under `src/`
- use TypeScript
- use Tailwind and shared UI primitives
- keep landing pages separate from the authenticated app shell
- keep business logic out of presentational components
- do not access Firestore directly from components
- centralize routing, constants, and configuration
- preserve multi-studio support from the start

These principles are explicitly part of the technical architecture and technical stories baseline.[file:181]

## App and landing page separation

The architecture separates public Studio landing pages from the shared authenticated app shell.

Rules:
- each Studio gets its own landing page directory under `src/app/`
- landing pages use shared marketing components plus Studio-specific content config
- the authenticated experience lives under `src/app/(app)/`
- `useStudio()` and Studio config are primarily for the authenticated shell, not for switching landing page content dynamically

This separation prevents the public marketing layer from becoming entangled with authenticated app behavior.[file:181]

## Routing conventions

Routing should follow these rules:
- public landing pages exist as Studio-specific app routes
- authenticated routes live inside `(app)`
- route files follow App Router conventions
- dynamic routes use `[param]`
- all route paths should be exported through `src/constants/routes.ts`
- login and register remain top-level public routes
- widget routes remain public where required

The documented route map includes placeholders for dashboard, programs, tools, events, cohorts, assignments, reports, notifications, settings, and admin paths.[file:181]

## Domain routing model

The architecture expects domain-based rewrites so each Studio domain resolves to its own landing page while preserving clean `/` URLs.

Working principle:
- coaching domain renders `src/app/coaching-studio/page.tsx`
- training domain renders `src/app/training-studio/page.tsx`
- recruitment domain renders `src/app/recruitment-studio/page.tsx`

For local development, developers can use the direct internal paths instead of relying on domain rewrites.[file:181]

## Layout foundation

T2 should establish:
- root layout at `src/app/layout.tsx`
- authenticated shell layout at `src/app/(app)/layout.tsx`

Root layout should:
- define global HTML/body wrappers
- load fonts
- wrap global providers
- support error boundaries and global app setup

Authenticated layout should:
- provide app shell structure
- host header/sidebar/main content regions
- prepare for auth-guard behavior
- pass Studio config context into child routes later

This layout work is the base that E1 and later app epics build on.[file:181]

## Module pattern

Each feature area should follow a consistent module structure:

```text
src/modules/[module-name]/
  components/
  hooks/
  types/
  utils/
  index.ts
```

Rules:
- function components only
- one primary component per file
- props typed in TypeScript
- module-specific logic stays in hooks/utils/services, not in render-heavy components
- shared UI should extend the common component library rather than reinvent primitives

This pattern is meant to keep feature work predictable and refactor-friendly.[file:181]

## Component rules

Frontend components should follow these rules:
- no direct Firestore imports in components
- no business logic embedded in UI rendering
- prefer presentational components with typed props
- use shared primitives and utility helpers
- keep naming conventions consistent

Naming conventions:
- components: `PascalCase.tsx`
- hooks: `useCamelCase.ts`
- services: `camelCase.service.ts`
- types: `camelCase.types.ts`
- utilities: `camelCase.ts`

These conventions are explicitly defined in the technical stories guidance.[file:181]

## State management

Use Zustand for lightweight global application state where needed.

Core stores to create:
- `auth.store.ts`
- `studio.store.ts`
- `notifications.store.ts`

Rules:
- initialize from server/auth state, not random local persistence
- reset state on logout where appropriate
- keep sensitive data out of browser stores beyond normal auth state
- do not overuse global state where local component state is enough

The goal is predictable shell-level state, not unnecessary complexity.[file:181]

## Service layer rules

All backend reads/writes must go through the service layer.

Rules:
- service files live under `src/services/`
- service functions are async and typed
- service functions accept `studioType` where tenant scoping is required
- catch and normalize backend errors
- throw typed service errors instead of raw SDK errors
- restrict Firebase SDK imports to `src/lib/firebase.ts` and `src/services/`

This is one of the most important architecture constraints in T2.[file:181]

## Constants and shared types

Create and centralize:
- role constants
- Studio type constants
- route constants
- site constants
- shared platform types

Important constants files:
- `src/constants/roles.ts`
- `src/constants/studio.ts`
- `src/constants/routes.ts`

The goal is to avoid string duplication and route drift across modules.[file:181]

## Shared hooks and utilities

Hooks and utilities should be centralized where they are cross-cutting and reused.

Examples already expected by the architecture:
- `useStudio`
- `useCurrentUser`
- `useUserContext`
- `useIsMobile`

Utilities should cover areas like:
- class merging
- date formatting
- general formatting
- validation
- storage helpers

Keep hooks focused, typed, and free from hidden business-side effects.[file:181]

## Form architecture

Forms should follow a shared approach rather than ad hoc implementation.

Pattern expectations:
- use `react-hook-form`
- use `zod` for validation
- create reusable form primitives
- keep validation logic structured and testable
- standardize error display and loading-state behavior

Suggested shared form components include:
- `FormInput`
- `FormSelect`
- `FormTextarea`
- `FormFileUpload`
- `FormSubmitButton`

This gives later auth, settings, and admin forms a clean consistent baseline.[file:181]

## Marketing content config

Landing page content should be config-driven by Studio.

Create:
- `src/modules/marketing/coaching-studio/content.config.ts`
- `src/modules/marketing/training-studio/content.config.ts`
- `src/modules/marketing/recruitment-studio/content.config.ts`

These files should supply:
- hero copy
- counters
- benefit cards
- CTA content
- SEO metadata
- social/contact links
- other Studio-specific landing-page content

This keeps shared marketing components reusable across all three Studio brands.[file:181]

## Engineering constraints

T2 should enforce these implementation boundaries:
- no backend access directly inside components
- no duplication of route strings across the app
- no mixing landing-page logic with authenticated-shell logic
- no feature module bypassing the shared architecture
- no hardcoded Studio switching hacks for public pages
- no unstable shortcuts that will force rework in E1+ epics

The purpose is to make all future work land on a stable architectural base.[file:181]

## Open inputs

These details may still evolve:
- exact shell visual design
- final iconography choices
- exact shared UI component set
- degree of provider composition in root layout
- some module boundaries as business features mature

Even if details evolve, the structure should remain stable enough that later epics do not need architectural rewrites.

## Delivery guidance

Treat T2 as one of the most important engineering epics.

If T2 is clean, Copilot suggestions, code reviews, onboarding, and later epic implementation all improve. If T2 is messy, every future epic becomes harder than it should be.[file:181]
