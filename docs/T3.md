# T3 — Deployment & DevOps

Status: Working markdown version for Copilot context.  
Canonical source: T3 technical foundation spec and Technical Stories Master.

## Objective

Establish the deployment model, environment strategy, and DevOps workflow for StudioVerse so the platform can be released in a controlled, repeatable, and production-ready way.

T3 should ensure the frontend is deployable via Vercel, the repository is integrated with deployment workflows, Firebase deployment is structured, environments are separated, domains and HTTPS are ready, builds are validated, and rollback is documented.[file:250][file:181]

## Scope

Epic T3 covers:
- Vercel setup
- GitHub integration
- environment-variable management across environments
- Firebase deployment configuration
- CI/CD workflows
- multi-environment setup
- domain mapping
- SSL/HTTPS readiness
- build validation
- rollback strategy

This epic is about release infrastructure, not business feature delivery.[file:250][file:181]

## Purpose

T3 transforms the project from a locally runnable codebase into a deployable product platform.

Without this layer, releases become manual, fragile, hard to validate, and hard to recover from. T3 introduces controlled delivery discipline across frontend and backend systems.[file:250]

## In scope

T3 should define and implement:
- frontend hosting on Vercel
- source-control-driven deployment behavior
- environment-specific configuration handling
- Firebase deploy commands and targets
- CI checks before merge/release
- dev / staging / prod separation
- custom domain connection path
- HTTPS validation
- repeatable build checks
- rollback runbooks

## Out of scope

Do not treat T3 as the place for:
- business feature implementation
- advanced observability dashboards
- deep analytics/product telemetry
- autoscaling optimization
- app store/mobile deployment
- feature-specific monitoring behavior beyond baseline release readiness

Those can come later.[file:250]

## Strategic role

T3 is the operational release layer for all later epics.

Public landing pages, app-shell routes, auth, dashboard work, and backend features all depend on the existence of a stable path from repository change to deployable output. T3 makes that path reliable and understandable.[file:250]

T3 depends on:
- T0 for repository and dev setup

It also depends on alignment around:
- branch workflow

## Vercel setup

Use Vercel as the frontend hosting and deployment platform.

Requirements:
- create/configure the Vercel project(s)
- connect the frontend codebase
- ensure Next.js App Router builds correctly on Vercel
- keep hosting setup aligned with the StudioVerse architecture

Acceptance baseline:
- frontend deploy succeeds
- hosted foundation is ready for live environments[ file:250]

## GitHub integration

Connect the repository to Vercel and release automation.

Expected behavior:
- repository activity should trigger deployment behavior
- source control should remain the release source of truth
- branch activity should map cleanly into environment behavior

The technical stories already align branch usage with deployment flow:
- `dev` for dev deployment
- `staging` for staging deployment
- `main` for production deployment[ file:181][file:250]

## Environment variables

Environment variables must be managed safely and separately for each environment.

Goals:
- different values for dev, staging, and prod
- no hardcoded sensitive config
- no accidental exposure of secrets
- alignment between frontend config and backend deployment targets

Important public variables include:
- `NEXT_PUBLIC_STUDIO_TYPE`
- `NEXT_PUBLIC_APP_ENV`
- `NEXT_PUBLIC_APP_BASE_URL`
- Firebase public config values
- `NEXT_PUBLIC_ENABLE_MOCK_SESSION`

Important rule:
- `NEXT_PUBLIC_` values are browser-safe only
- secrets must remain server-side in Vercel/Firebase environment config[ file:181][file:250]

## Vercel environment mapping

Per the technical stories, Vercel environment setup should support the three standard environments:
- dev
- staging
- prod

Working expectations:
- each environment should receive the correct Firebase project config
- `NEXT_PUBLIC_APP_ENV` should clearly reflect the environment
- app-shell Studio context should remain configurable via `NEXT_PUBLIC_STUDIO_TYPE`
- `.env.example` should stay in sync with required variables[ file:181]

## Firebase deploy config

Backend deployment must be structured and repeatable through Firebase CLI configuration.

Expected targets include:
- Firestore rules
- Firestore indexes
- Storage rules
- Functions

Working deploy patterns:
- full deploys per environment
- rules-only deploys
- functions-only deploys
- environment selection through Firebase project aliases

This should all remain documented and source-controlled rather than tribal knowledge.[file:181][file:250]

## CI/CD

Set up automated validation and deployment workflows.

Minimum workflow expectations:
- PR CI should install dependencies, run type-check, run lint, and run build
- failing checks should block merge confidence
- production merges should support automated deploy steps for backend targets where configured

The technical stories specifically outline GitHub Actions workflows for CI, functions deploy, and rules deploy, making CI/CD a formal part of the expected release model.[file:181]

## Multi-environment model

The platform should operate with three clearly separated environments:
- dev
- staging
- prod

Environment matrix expectations:
- dev uses `studioverse-dev`
- staging uses `studioverse-staging`
- prod uses `studioverse-prod`

Validation goals:
- correct Firebase project per environment
- isolated auth/data/storage
- no accidental bleed between dev and prod
- environment behavior documented in README or equivalent runbook[ file:181]

## Domain mapping

Prepare and validate custom domain support for each Studio.

Production domains expected by the technical stories:
- `coachingstudio.io`
- `trainingstudio.io`
- `recruitmentstudio.io`

Domain rules:
- configure registrar DNS to Vercel
- choose and enforce canonical host style
- redirect alternate host consistently
- ensure domain mapping works with the Next.js rewrite model defined in T2

Domain mapping alone is not enough; it must work together with the routing/rewrite setup.[file:181]

## SSL / HTTPS

HTTPS must be active for deployed public access.

Validation checklist:
- SSL active on all production domains
- auto-renewal handled by Vercel
- HTTP redirects to HTTPS
- no browser certificate warnings
- no mixed-content issues on loaded pages

The T3 spec treats this as required operational validation even though Vercel automates most certificate provisioning.[file:250][file:181]

## Build validation

Build validation must be part of the release process.

Goals:
- confirm builds succeed cleanly
- catch TypeScript failures before release
- catch lint failures before release
- confirm different Studio variants build successfully
- document relevant build scripts

The technical stories specifically call for validating builds for coaching, training, and recruitment Studio contexts rather than assuming one build path covers all cases.[file:181]
