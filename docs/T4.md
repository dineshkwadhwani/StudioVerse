# T4 — Security & Access Control

Status: Working markdown version for Copilot context.  
Canonical source: T4 technical foundation spec and Technical Stories Master.

## Objective

Define and implement the baseline security posture and access-control framework for StudioVerse.

T4 should ensure roles are clearly defined, protected routes can be guarded, Firestore rules enforce least privilege, tenant and relationship boundaries are respected, Functions validate requests safely, sensitive actions are traceable, secrets are protected, and PII is handled carefully.[file:254][file:181]

## Scope

Epic T4 covers:
- role model
- auth guard patterns
- Firestore security rules
- multi-tenant data isolation
- secure Firebase Function practices
- input validation standards
- logging of security-relevant actions
- rate limiting approach
- secure environment variable handling
- PII handling baseline

This epic is foundational security work, not end-user feature development.[file:254]

## Purpose

T4 is the trust layer for the whole platform.

Because StudioVerse handles user identity, assignments, reports, progress data, company-level access, and role-based workflows, weak security would affect every later epic. T4 exists so later features are built inside a secure structure instead of trying to patch security afterward.[file:254]

## In scope

T4 should establish:
- standard role definitions
- protected route behavior
- database-layer access controls
- tenant and relationship-based data boundaries
- trusted backend validation patterns
- logging expectations for sensitive actions
- abuse protection thinking for high-risk flows
- clear rules for secret handling
- privacy-conscious treatment of user data

## Out of scope

Do not treat T4 as:
- a full legal compliance program
- external penetration testing
- enterprise SSO/identity suite implementation
- full SIEM/SOC rollout
- privacy policy drafting
- end-user security training content

Those may come later, but this epic defines the secure technical baseline first.[file:254]

## Strategic principles

Security in StudioVerse should follow these principles:
- security by design
- least privilege
- multi-tenant isolation
- no trust in client-only state
- backend validation for privileged actions
- traceability for sensitive operations
- minimize unnecessary data exposure
- keep public surfaces especially light on personal data

These principles are explicit in the formal T4 guidance and should shape implementation choices everywhere.[file:254]

## Role model

The platform should standardize on four roles:
- `superadmin`
- `company`
- `professional`
- `individual`

The technical stories map these into Studio-specific labels such as coaching firm, coach, and coachee while keeping the underlying role identifiers stable across the codebase.[file:181]

## Role implementation rules

Working implementation expectations:
- roles live in `users.roles` as a string array
- `userContexts` acts as the RBAC oracle
- frontend role data should come from `useUserContext()`
- role constants should live in `src/constants/roles.ts`
- TypeScript role types should be centralized and reused

Multi-role users are allowed, and access should be additive unless explicitly overridden by stronger restrictions. `superadmin` remains the highest-privilege role.[file:181]

## Auth guards

Protected application areas should use layered access control.

Recommended guard layers:
- middleware checks authentication presence for `(app)` routes
- layout-level logic checks role-appropriate access using `userContext`
- optional client hooks can help with UX, but should not be the only guard

Goal:
- unauthenticated users are redirected to `/login`
- authenticated users without required access are redirected safely
- authorization logic is centralized, not duplicated page by page[ file:181][file:254]

## Route protection expectations

Working route examples from the technical stories:
- `/admin/*` → `superadmin` only
- `/cohorts/*` → `company` and `professional`
- role-specific dashboards should resolve by user role
- authenticated shared areas may be visible to multiple roles, but data inside them must still be filtered

This means route access and data access are related, but not identical. Even when a route is accessible, underlying data still needs filtering and rules enforcement.[file:181]

## Firestore rules

Firestore rules must enforce the real security boundary at the database layer.

Core rules:
- all reads and writes require authentication
- least privilege should be the default
- users should not access unrelated records
- `studioType` must participate in tenant scoping
- write permissions should be narrower than read permissions where appropriate

The technical stories explicitly expect full rules coverage across all 25 collections, not just a placeholder deny-all baseline.[file:254][file:181]

## Collection security patterns

Important collection-level expectations include:
- `users`: users can read/write their own document; superadmin has elevated visibility
- `userContexts`: readable by the owner, writable by Functions only
- `assignments`: readable by superadmin, owning professional, or assigned individual
- `reports`: readable only by authorized actors; written through trusted backend paths
- `auditLogs`: readable by superadmin only, written by backend processes
- `toolSubmissions`: constrained to the submitter and authorized upstream roles

These examples are enough to guide implementation patterns even before every final rule is written line by line.[file:181]

## Data isolation

StudioVerse must enforce strong logical separation between tenants and unrelated users.

Isolation model should include:
- `studioType` scoping across tenant collections
- relationship-based scoping such as `professionalId` and `companyId`
- no access to unrelated tenant data by default
- superadmin visibility only within their allowed Studio context unless explicitly designed otherwise

The formal T4 document and technical stories both treat multi-tenant isolation as a core structural requirement, not an optional optimization.[file:254][file:181]

## Application-layer enforcement

Application code should reinforce, but never replace, database security.

Rules:
- all service-layer list queries should include `studioType`
- professional-scoped queries should include `professionalId` where relevant
- Studio context should come from trusted app config, not arbitrary URL/user input
- UI components must never assume visibility without service-layer and rules-layer support

This prevents accidental cross-tenant or cross-user leakage even before Firestore rules reject bad requests.[file:181]

## Secure Functions

Firebase Functions are the trusted execution layer for privileged business logic.

Every sensitive Function should:
- verify authentication
- verify role/authorization from trusted backend context
- validate all inputs
- reject mismatched or unsafe tenant context
- log critical execution details
- return typed safe errors instead of raw stack traces

The technical stories recommend a secure wrapper pattern so authorization and validation are standardized across Functions.[file:181][file:254]

## Input validation

Validation should happen in both frontend and backend, but for different reasons.

Rules:
- frontend validation improves UX
- backend validation is mandatory for trust
- use schema-based validation patterns
- reject malformed, unexpected, or unauthorized payloads early
- never treat client validation as sufficient security

This is one of the clearest implementation boundaries in the T4 guidance.[file:254][file:181]

## Logging

Critical security-relevant actions should be traceable.

Examples of events worth logging:
- privileged backend actions
- access-sensitive operations
- permission failures
- important state changes triggered by secure workflows
- security-relevant Function execution

Logs should support troubleshooting, accountability, and future audit visibility without becoming a raw dump of sensitive data.[file:254]

## Rate limiting

T4 should define a baseline approach for abuse-sensitive actions.

Typical candidate flows:
- login or OTP-like flows in later epics
- public or semi-public submission endpoints
- repeated calls to sensitive backend operations
- chatbot or widget abuse-sensitive endpoints if introduced later

Even where full implementation is deferred, the technical approach should be documented so later features do not invent it ad hoc.[file:254]

## Secure environment handling

Secrets must stay out of source code and out of browser-exposed config.

Rules:
- no secrets in committed files
- no secrets in `NEXT_PUBLIC_*`
- privileged values belong in Vercel server env or Firebase Functions config
- frontend should only receive values intended for browser use
- environment handling should stay aligned with T0 and T3 conventions

This is a shared responsibility across setup, deployment, and security work.[file:254][file:181]

## PII handling

The platform should follow a privacy-conscious baseline.

Rules:
- identify what counts as PII in the platform
- collect only what is needed
- expose only what is needed for the current role and workflow
- avoid unnecessary PII on public surfaces such as landing pages
- design later features with data minimization in mind

The formal T4 spec explicitly warns against careless exposure or overcollection of personal data.[file:254]

## Risks to avoid

T4 is intended to reduce:
- overly permissive Firestore rules
- inconsistent role interpretation
- trusting client state for secure actions
- weak input validation
- overexposure of sensitive data
- missing traceability for important security events

Mitigation approach:
- define roles early
- keep business logic in Functions where needed
- enforce least privilege in rules
- validate all inputs
- log critical actions
- minimize unnecessary personal data exposure[ file:254]

At the end of T4, the project should have:
- documented role model

- secure Functions standards

- role constants are defined and standardized
- `userContext`-based role access model is documented
- protected route pattern exists
- unauthenticated access can be blocked cleanly
- Firestore rules are defined and deployable
- all sensitive collections are not openly accessible
- tenant and relationship isolation rules are documented
- service-layer filtering aligns with security model
- secure Function standards are defined
- validation standards exist for frontend and backend
- logging expectations are documented
- rate limiting baseline is identified
- secret-handling guidance is documented
- PII handling baseline is documented

## Open inputs

These areas may still evolve:
- exact final Firestore rule expressions per collection
- exact middleware/layout implementation split
- final audit log schema
- final rate-limiting mechanism
- exact `userContext` document shape
- final incident logging/monitoring provider integration

Even if details evolve, the least-privilege and tenant-isolation principles should remain fixed.

## Delivery guidance

Treat T4 as a hard constraint on later implementation, not a side note.

If T4 is applied well, later epics naturally follow safe patterns. If it is weak or ignored, every future feature becomes harder to trust and more expensive to correct.[file:254][file:181]
