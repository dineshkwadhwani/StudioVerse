# T5 — Studio Config Architecture

Status: Working markdown version for Copilot context.  
Canonical source: Technical Stories Master status summary and epic goal.  
Important note: detailed T5 user stories are still pending in the master technical stories document.

## Objective

Define and implement the per-Studio configuration system that controls terminology, feature flags, branding, and behavior across Coaching Studio, Training Studio, and Recruitment Studio.

T5 exists because StudioVerse is one codebase serving three Studio deployments, so Studio-specific differences should be configuration-driven instead of hardcoded into feature logic.[file:181]

## Scope

Epic T5 should cover:
- per-Studio terminology
- branding configuration
- feature flags
- app-shell labels and copy
- Studio-aware runtime behavior
- config typing and validation
- clear separation between app config and landing-page content config

This is foundational architecture work, not a business feature epic.[file:181]

## Purpose

T5 is the configuration layer that makes the one-codebase/three-Studio model maintainable.

Without a dedicated config architecture, the codebase will drift into scattered `if studioType === ...` logic, duplicated components, inconsistent labels, and harder maintenance. T5 should prevent that by centralizing Studio-specific behavior behind typed config objects.[file:181]

## Relationship to other epics

T5 sits on top of several earlier technical decisions:
- T0 defines environment conventions such as `NEXT_PUBLIC_STUDIO_TYPE`
- T2 defines the app-shell hooks and `src/config/studios/` structure
- T3 defines separate Vercel deployments with Studio-specific environment values
- T4 depends on stable terminology and role-label mapping in the app shell

That means T5 is not isolated; it is the glue that makes the shared architecture behave correctly per Studio.[file:181]

## Core principle

The app should share one implementation wherever possible and vary behavior through configuration.

Use configuration for:
- labels
- naming
- feature visibility
- support contacts
- dashboard wording
- branding tokens
- Studio-specific links

Do not fork components or duplicate routes unless a Studio genuinely needs different functionality rather than different presentation or terminology.[file:181]

## Studio types

The configuration system should support these three Studio types:
- `coaching`
- `training`
- `recruitment`

These values already appear in the broader architecture and environment setup and should remain the canonical Studio identifiers across config, services, and runtime behavior.[file:181]

## Config boundary

T5 should govern the authenticated app shell and shared runtime behavior.

Important separation:
- `studio.config.ts` handles app-shell config such as terminology, labels, feature switches, and runtime Studio behavior
- `content.config.ts` handles public landing-page copy, SEO content, and marketing sections

This separation is already implied in T2 and should remain strict so marketing copy changes do not interfere with app-shell behavior.[file:181]

## Suggested config files

The T2 folder structure already defines the intended config location:

```text
src/config/studios/
  studio.config.ts
  coaching.config.ts
  training.config.ts
  recruitment.config.ts
```

This is the right place for Studio runtime config because it keeps Studio-specific behavior centralized and discoverable.[file:181]

## Base config model

Use a shared base type plus one config file per Studio.

Suggested pattern:
- `studio.config.ts` exports types, shared helpers, and `getStudioConfig()`
- each Studio config exports a typed object
- config objects satisfy one shared `StudioConfig` interface
- runtime selection resolves from trusted Studio context, not arbitrary user input

This keeps both TypeScript safety and operational clarity strong.[file:181]

## Suggested StudioConfig shape

A practical `StudioConfig` shape should include:
- `studioType`
- `displayName`
- `shortName`
- `roleLabels`
- `terminology`
- `branding`
- `featureFlags`
- `support`
- `navigation`
- `dashboard`
- `legal`
- `externalLinks`

The exact field list can evolve, but the central idea is that all Studio differences should flow from a typed object rather than scattered literals.[file:181]

## Role labels

The platform role identifiers should stay stable:
- `superadmin`
- `company`
- `professional`
- `individual`

But their user-facing labels should come from Studio config, for example:
- coaching: Coaching Firm / Coach / Coachee
- training: Training Company / Trainer / Learner
- recruitment: Recruitment Agency / Recruiter / Candidate

This preserves a stable RBAC model while allowing Studio-appropriate language in the UI.[file:181]

## Terminology mapping

Terminology should be config-driven for any user-facing word that differs by Studio.

Typical examples:
- professional label
- individual label
- company label
- dashboard headings
- assignment wording
- program/tool/event copy where naming differs
- CTA labels inside the authenticated app shell

This helps Copilot and developers avoid hardcoding Coaching-specific language into shared components.[file:181]

## Branding config

Branding should also be config-driven.

Recommended branding fields:
- app name
- logo path
- favicon path
- primary color token
- accent color token
- email sender/display name
- support email
- social/profile links if needed in the app shell

Branding should influence presentation, not business logic.[file:181]

## Feature flags

Feature availability should be configurable per Studio.

Examples of flags that may belong here:
- events enabled
- marketplace enabled
- widget enabled
- reports enabled
- payments enabled
- admin tools visibility
- beta features

Feature flags should be evaluated centrally so shared components and navigation can hide or show behavior predictably.[file:181]

## Navigation config

Navigation should be derived from Studio config plus role access.

Recommended approach:
- define the full route map once
- filter visible navigation items by role and feature flags
- allow labels and optional ordering tweaks through Studio config
- avoid per-Studio nav duplication unless the route structure truly differs

This keeps navigation consistent while still allowing Studio-specific presentation.[file:181]

## Runtime source of truth

Studio selection should come from trusted runtime context.

Expected sources:
- deployed environment via `NEXT_PUBLIC_STUDIO_TYPE` for the app shell
- local development via the same env var
- domain/path mapping for landing pages, which should remain separate from app-shell config logic

Do not allow Studio identity for protected app behavior to come from arbitrary query params or editable client state.[file:181]

## Hook and store integration

The T2 architecture already anticipates this integration:
- `useStudio()` should expose the active Studio config
- `studio.store.ts` should hold `studioType`, `config`, and `terminology`
- shared components should consume config through hook/store access rather than importing per-Studio files directly

That pattern keeps the rest of the codebase clean and prevents config leakage into random modules.[file:181]

## Validation rules

T5 should require validation and type safety around config.

Rules:
- every Studio config must satisfy the same TypeScript interface
- required keys must exist for all Studios
- no silent fallback to undefined labels
- default values should be explicit, not accidental
- config resolution failures should fail loudly in development

This is important because configuration mistakes in a multi-Studio system are easy to miss until runtime.[file:181]

## Implementation rules

Use these working rules for the codebase:
- no inline Studio-specific labels inside shared components
- no scattered `if/else` Studio logic when config lookup will do
- keep role identifiers stable and labels configurable
- keep feature flags centralized
- keep landing-page marketing content outside T5 config
- treat Studio config as app architecture, not ad hoc constants

These rules are the practical heart of T5.[file:181]

## Example pattern

A healthy usage pattern looks like this:

```typescript
const { config } = useStudio()

<h1>{config.dashboard.homeTitle}</h1>
<p>Create a new {config.roleLabels.professional}</p>
```

A poor pattern looks like this:

```typescript
const label =
  studioType === 'coaching'
    ? 'Coach'
    : studioType === 'training'
    ? 'Trainer'
    : 'Recruiter'
```

The goal is to keep Studio variation declarative and centralized.[file:181]

## Deliverables

At the end of T5, the project should have:
- typed Studio config model
- one config file per Studio
- centralized config resolver
- terminology mapping
- role-label mapping
- branding tokens and support metadata
- feature-flag structure
- `useStudio()` / store integration
- documented boundary between app config and landing-page content config

These deliverables would complete the configuration foundation implied by the one-codebase architecture.[file:181]

## Acceptance checklist

- `src/config/studios/studio.config.ts` defines the shared types and resolver
- `coaching.config.ts`, `training.config.ts`, and `recruitment.config.ts` exist
- all three configs satisfy one shared `StudioConfig` type
- role identifiers are stable, labels are config-driven
- navigation and feature flags are config-driven where appropriate
- shared components do not hardcode Studio-specific labels
- `useStudio()` returns typed config data
- landing pages continue using separate `content.config.ts` files
- no business logic depends on marketing config
- config mistakes fail clearly in development

## Delivery guidance

Treat T5 as the anti-fragmentation layer for StudioVerse.

If T5 is done well, the shared codebase stays clean, configurable, and scalable. If it is skipped, Studio-specific divergence will spread across components, services, and UI copy in ways that become expensive to unwind later.[file:181]
